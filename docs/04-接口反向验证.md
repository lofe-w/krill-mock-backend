# 接口反向验证 · 用三方 API 文档核对当前三接口是否够用

> 方法：把前端《数据对接 API 文档》的**六大模块**逐字段映射到我方 `value/records/series` 三接口，
> 标"支持/缺口"。结论先行：**三接口覆盖全部六模块**，前端每屏 2~3 次调用即可拼齐

## 缺口汇总

**数据类(补几个 key 即可，小)**
- A. 导航聚合量：`船舶.航行.航程/航行距离/航行时间`(累计/派生)、`船舶.航行.航行状态`(离散)。
- B. `工厂.综合楼.总耗电量/总耗水量`、`工厂.虾油线.生产数据.磷脂回收率`(派生)、仓库温湿度。
- C. 窗口聚合 `{当天/当月/当前渔季/年度}`：当前前端按 start/end 各查一次做"末−首"。如嫌调用多，可给 `series` 加可选 `窗口` 参数一次返四值（便利项，非必需）。

**结构类（有意，不改我方）**
- D. 溯源 追溯/检验在环节内（甲方口径）vs 前端顶层 → 前端适配或加薄映射层。

**工程横切（非数据，production 必需）**
- E. **鉴权**：当前无。→ .env 静态 Token + Bearer 校验。
- F. **视频流**：地址已返回，但需流媒体服务(MediaMTX)循环推 N 段片源 → N 地址；浏览器播放需 HLS。
- G. **CORS**：浏览器跨域调用 → 需开启。

## 结论
- ✅ **三接口(value/records/series)覆盖全部六模块**；前端每屏 2~3 次调用拼齐 → **step① 组合层确认不做**。
- 待补：缺口 A/B 的少量 key（小）；横切 E/F/G（鉴权/视频/CORS，工程化收尾）；C/D 是便利与适配，可选。
