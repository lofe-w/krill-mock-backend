# 配置与取数契约 · 注册表 Schema（机器可加载层）

| 项 | 内容 |
|---|---|
| 文档地位 | `数据字典.md` 下沉一层：把每条 key **声明为机器可加载的配置**，并定义"一次查询如何被解析"的取数契约。**直接对应 `config/*.yaml` 与 resolver 行为** |
| 上游 | `数据字典.md`（全量 key 实例）、`domain.md`（四形状定义） |
| 版本 | v1.0 / 2026-06-23 |

> **下沉即验证（本步结论）**：把四形状 + 事件 + 投影 + 成熟度逐一写成 YAML，schema 基本够用；但暴露**一个真实缺口**——`数据字典.md` §0 的"每 key 一条声明"装不下**跨 key 的不变式/派生**（物料守恒、上界、份额派生都牵涉多个 key）。故本文新增一个与注册表并列的**约束/派生层**（§3），这是单 key schema 之外必须有的第二张表。

---

## 1. 总体结构

```
config/
├── registry/            # 注册表：每个 key 一条声明（按业务域分文件）
│   ├── 船舶.yaml
│   ├── 工厂.yaml
│   ├── 溯源.yaml
│   └── 主数据.yaml
├── constraints.yaml     # ★约束/派生层（跨 key，见 §3）
├── overrides.yaml       # 覆盖/冻结/兜底
└── sources.yaml         # 真值源连接信息（密钥走环境变量，不入库）
```

注册表条目通用头（四形状共有）：

```yaml
- key:      <全限定名>
  形状:     A | B | C | D
  成熟度:   人工固定 | 规则生成 | 真值采集
  出处:     甲方 <节号>      # 可选，便于追溯
```

形状决定**请求/响应的轴**，成熟度决定**值从哪来**——二者正交（`domain.md` §2 / §7）。

## 2. 四形状的声明 Schema（含代表性实例）

### 2.1 A `(key, value)` — 单一站立

```yaml
- key: 船舶.信息
  形状: A
  成熟度: 人工固定
  value: { IMO: "9849332", 船名: 深蓝号 }

- key: 船舶.桁杆泵吸系统.设计能力
  形状: A
  成熟度: 人工固定
  value: { 日产量: 500, 年产量: 100000, 渔季产量: 100000, 单位: 吨 }
```

### 2.2 B `(key, filter, value)` — 可选站立

```yaml
- key: 仓储                      # 形状 B/A 待 Q1 确认；暂记 B
  形状: B
  成熟度: 人工固定
  filter: [批号, 仓库类别]        # 开放实例集维度
  条目:
    - filter: { 批号: SLHDX-2026-001, 仓库类别: 深蓝号冻虾舱 }
      value: { 货物名称: 冻虾, 数量: 3000, 规格: "10kg/箱", 保质期: "2027-06-18",
               生产日期: "2026-06-15", 入库日期: "2026-06-16", 出库日期: "", 冷藏状态: 正常 }

- key: 人员
  形状: B
  成熟度: 人工固定
  filter: [所属]
  条目:
    - filter: { 所属: 虾粉线 }
      value:
        姓名: 王五
        电话: "13700137000"
        职务:   [ {time: "2026-01-01 00:00:00", value: 操作工}, {time: "2026-04-01 00:00:00", value: 班长} ]
        所在位置: [ {time: "2026-01-01 00:00:00", value: 厂区} ]
        是否在岗: [ {time: "2026-06-18 08:00:00", value: true} ]
```

> 时序属性是 value **内嵌**的升序变更数组（不走 C）。

### 2.3 C `(key, time, value)` — 单一时序

```yaml
# 规则生成·瞬时
- key: 船舶.冻虾线.冷库温度
  形状: C
  成熟度: 规则生成
  量语义: 瞬时
  网格: "0 * * * *"             # 预设，待 Q6
  单位: ℃
  规则: { kind: 有界瞬时, 基线: -25, 幅度: 2, 区间: [-30, -18] }

# 规则生成·累计
- key: 船舶.捕捞.累计产量
  形状: C
  成熟度: 规则生成
  量语义: 累计
  网格: "*/3 * * * *"
  单位: 吨
  规则: { kind: 累计, 速率基线: 0.5, 起点: 渔季起点 }

# 真值采集（成熟度不同，形状仍是 C）
- key: 工厂.天气.温度
  形状: C
  成熟度: 真值采集
  量语义: 瞬时
  单位: ℃
  来源: { kind: open-meteo, 落库: sqlite, 字段: temperature_2m }   # 连接见 sources.yaml
```

### 2.4 D `(key, filter, time, value)` — 可选时序 / 离散事件

D 有两种 value 语义（`domain.md` §2.3），用 `语义` 字段区分：

```yaml
# 语义=采样：某批次的冷链温度曲线
- key: 冷链温度
  形状: D
  语义: 采样
  成熟度: 规则生成
  filter: [批号, 段]
  量语义: 瞬时
  单位: ℃
  规则: { kind: 有界瞬时, 基线: -20, 幅度: 2 }

# 语义=事件：一次生产作业的输入输出包（5.3）
- key: 生产作业.虾油
  形状: D
  语义: 事件                     # 时间=事件完成时刻，非采样网格
  成熟度: 规则生成
  filter: [作业批次]
  事件包:
    基准: { 物料: 原料全脂虾粉, 字段: 输入重量, 规则: {kind: 有界瞬时, 基线: 700, 幅度: 50} }
    inputs:
      - { 物料: 原料全脂虾粉, 字段: [输入重量, 磷脂含量, 脂肪含量] }
      - { 物料: 无水酒精,    字段: [输入重量, 浓度] }
      - { 物料: 电力 }; - { 物料: 蒸汽 }; - { 物料: 燃气 }; - { 物料: 人力资源 }
    outputs:
      - { 物料: 虾油,        字段: [输出重量, 磷脂含量], 派生: 见 constraints.yaml#物料守恒.虾油 }
      - { 物料: 脱脂虾粉,    字段: [输出重量, 磷脂含量], 派生: 同上 }
      - { 物料: 回收无水酒精, 字段: [输出重量, 浓度] }
```

> 事件包的 outputs **不独立取值**，由"基准 × 份额"派生（见 §3），以恒满足物料守恒。

## 3. ★约束/派生层 `constraints.yaml`（跨 key，单 key schema 装不下）

> **本步发现的缺口**：物料守恒、产量上界、份额派生都**牵涉多个 key**，无法塞进任一 key 的条目。故独立成层，由 resolver 在求值时套用。

```yaml
# INV-1/2/6 物料守恒 + 得率派生（虾油作业）
物料守恒.虾油:
  基准: 生产作业.虾油#inputs.原料全脂虾粉.输入重量
  份额:                                   # Σ份额 ≤ 1，平滑噪声归一
    生产作业.虾油#outputs.虾油.输出重量:        { kind: 份额, 基线: 0.18 }
    生产作业.虾油#outputs.脱脂虾粉.输出重量:    { kind: 份额, 基线: 0.78 }
  得率: { key: 工厂.虾油线.生产数据.虾油得率, 定义: 虾油输出 / 全脂虾粉输入 }   # INV-2 派生

# INV-1 捕捞量 ≥ Σ各线投料
物料守恒.捕捞:
  基准: 船舶.捕捞.累计产量
  下游份额: [ 船舶.虾粉线.流量秤.当天累计, 船舶.冻虾线.流量秤.当天累计, 船舶.虾肉线.流量秤.当天累计 ]
  约束: Σ下游 ≤ 基准

# INV-5 实际产量 ≤ 设计能力（上界 clamp）
上界.虾粉线: { 受约束: 船舶.虾粉线.打包.当天累计重量, 上界: 船舶.虾粉线.设计能力.日产量 }

# INV-4 冷链温度链 + 中心温度 ≤ 冷库温度
温度链.冻虾:
  序: [捕捞≤加工≤冷链≤仓储≤检测]            # 时间因果
  约束: 检测#冻块中心温度 ≤ 冷链温度#对应段
```

> 实现策略（A 结构性派生优先于 B 钳制）见《造数引擎》§5。依赖图必须无环（INV-3；回收料按新批次入账、不回指上游）。

## 4. 溯源投影声明（B + 引用，不复制）

```yaml
- key: 溯源
  形状: B
  成熟度: 人工固定           # Fake 阶段配置喂；结构 = 回溯产出
  filter: [产品批号]
  环节顺序:                  # 按产品类别决定路径（路径长短不同）
    全脂虾粉: [磷虾捕捞, 船载加工, 冷链运输, 岸基仓储, 产品检测]
    虾油:     [磷虾捕捞, 船载加工, 冷链运输, 岸基仓储, 虾油提取线, 成品检测]
    虾油胶囊: [磷虾捕捞, 船载加工, 冷链运输, 岸基仓储, 虾油提取线, 封装, 成品检测]
  条目:
    - filter: { 产品批号: QZXF-001 }
      value:
        类别: 全脂虾粉
        溯源链:                # 各环节用引用，不复制别处数据
          船载加工: { 引用: 船舶.虾粉线.打包.当天累计重量, 批次: CZJG-001 }
          冷链运输: { 引用: 冷链温度, filter: { 批号: CZJG-001 } }
        产品检验: { 引用: 检测, filter: { 产品: 全脂虾粉, 批次: QZXF-001 } }
        原料来源追溯: { 捕捞海域: {经度: 120.5, 纬度: 35.2}, 捕捞方式: 拖网, 原料溯源: null }   # 可递归→上游批号
```

## 5. 取数解析契约（resolver 行为）

一次查询 `q = (key [, filter] [, time])` 的解析顺序：

```
1. spec = registry[key]            # 取声明；无 → 404
2. 形状校验：A 不收 filter/time；B 收 filter；C 收 time；D 收 filter+time
3. override 命中(overrides.yaml)？ → 返回覆盖/冻结/兜底值
4. 按成熟度分派取值：
     人工固定 → 读 config 条目（B/D 按 filter 过滤）
     规则生成 → 造数引擎求值（C/D 采样按网格；D 事件按 constraints 派生）
     真值采集 → provider 取数（sqlite / 远程时序库 / RTSP 地址）
5. 套用 constraints.yaml 中涉及本 key 的派生/约束（求值期，同一 t 递归、记忆化）
6. 按形状组装响应：A=值；B=命中条目；C=带 time 序列；D=带 filter+time 序列 / 事件包
```

- 统一查询入口，**按形状自然分派**（实现层统一，模型层不统一——`domain.md` §8）。
- 协议字段（key/filter/time/value/status）英文；业务名与 value 内部键中文。
- 单位拼接、图表结构归前端（关注点分离）。
