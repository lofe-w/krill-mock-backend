# 配置与取数契约 · 注册表 Schema（机器可加载层）

| 项 | 内容 |
|---|---|
| 文档地位 | `数据字典.md` 下沉一层：把每条 key **声明为机器可加载的配置**，并定义"一次查询如何被解析"。**直接对应 `config/*.yaml` 与 resolver 行为** |
| 上游 | `数据字典.md`（全量 key 实例）、`domain.md` v2.1（三表 A/B/C） |
| 版本 | v1.3 / 2026-07-16（D5/D6：C 表仅保留叶子 key、对外只 keys 寻址；series 用 `window` 表达时间+点数，支持全局或逐 key；§5.1 固定点数落地） |

> **下沉即验证（本步结论）**：把三表 + 投影 + 成熟度逐一写成 YAML，schema 够用；但暴露**一个真实缺口**——`数据字典` 的"每 key 一条声明"装不下**跨 key 的不变式/派生**（物料守恒、上界都牵涉多个 key）。故新增与注册表并列的**约束/派生层**（§3），这是单 key schema 之外必须的第二张表。

---

## 1. 总体结构

```
config/
├── registry/            # 注册表：每个 key 一条声明（按业务域分文件）
│   ├── 船舶.yaml  ├── 工厂.yaml  ├── 溯源.yaml  └── 主数据.yaml
├── constraints.yaml     # ★约束/派生层（跨 key，见 §3）
├── overrides.yaml       # 覆盖/冻结/兜底
└── sources.yaml         # 真值源连接信息（密钥走环境变量，不入库）
```

注册表条目通用头（三表共有）：

```yaml
- key:      <全限定名>
  表:       A | B | C
  成熟度:   人工固定 | 规则生成 | 真值采集
  出处:     甲方/前端 <节号>   # 可选
```

表决定**请求/响应的轴**，成熟度决定**值从哪来**——二者正交（`domain.md` §2 / §7）。

## 2. 三表的声明 Schema（含代表性实例）

### 2.1 A `(key, value)` — 单一站立

```yaml
- key: 船舶.信息
  表: A
  成熟度: 人工固定
  value: { IMO: "9849332", 船名: 深蓝号 }

- key: 船舶.桁杆泵吸系统.设计能力
  表: A
  成熟度: 人工固定
  value: { 日产量: 500, 年产量: 100000, 渔季产量: 100000, 单位: 吨 }
```

### 2.2 B `(key, filter, value)` — 可选站立

```yaml
- key: 仓储                      # B/A 待 Q1 确认；暂记 B
  表: B
  成熟度: 人工固定
  filter: [仓库类别, 产品批号]     # 开放实例集
  条目:
    - filter: { 仓库类别: 深蓝号冻虾舱 }
      value: { 货物名称: 冻虾, 批号: SLH2606AKF01, 数量: 3000, 规格: "10kg/箱",
               保质期: "2027-06-18", 生产日期: "2026-06-15", 入库日期: "2026-06-16", 出库日期: "", 冷藏状态: 正常 }

- key: 人员
  表: B
  成熟度: 人工固定
  filter: [所属]
  条目:
    - filter: { 所属: 虾粉线 }
      value:
        姓名: 王五
        电话: "13700137000"
        职务:   [ {time: "2026-01-01 00:00:00", value: 操作工}, {time: "2026-04-01 00:00:00", value: 班长} ]
        所在位置: [ {time: "2026-01-01 00:00:00", value: 厂区} ]
        是否在岗: [ {time: "2026-06-18 08:00:00", value: true} ]
```

> 时序属性是 value **内嵌**的升序变更数组（不走 C）。

### 2.3 C `(key, time, value)` — 时序

```yaml
# 规则生成·瞬时（按舱冷库温度——注意是按舱，非按批号）
- key: 船舶.冻虾舱.温度
  表: C
  成熟度: 规则生成
  量语义: 瞬时
  网格: "0 * * * *"             # 预设，待 Q6
  单位: ℃
  规则: { kind: 有界瞬时, 基线: -18, 幅度: 1.5, 区间: [-25, -15] }

# 规则生成·累计（窗口量 = 末−首）
- key: 船舶.捕捞.累计产量
  表: C
  成熟度: 规则生成
  量语义: 累计
  网格: "*/3 * * * *"
  单位: kg
  规则: { kind: 累计, 速率基线: 0.5, 起点: 渔季起点 }
  不变式: [INV-1, INV-5]

# 真值采集（成熟度不同，表仍是 C）
- key: 工厂.天气.温度
  表: C
  成熟度: 真值采集
  量语义: 瞬时
  单位: ℃
  来源: { kind: open-meteo, 落库: sqlite, 字段: temperature_2m }   # 连接见 sources.yaml
```

> **`规则.kind` 取值**：限于《造数引擎》§4.0 速查表登记的 4 个 `kind`（`有界瞬时`/`累计`/`离散状态`/`航迹`）；趋势/季节/多序列是其**参数**，跨 key 约束走 `constraints.yaml`（§3）——配置时勿臆造新 kind。

> ★**C 表只声明叶子 key**。每个可查询数据项都是一条独立 C key，例如 `船舶.海况.海水温度`。父级如 `船舶.海况` 不需要在配置中声明，三端点会按同表父系前缀动态展开后代叶子 key。这样消除了"有些数据传 metrics、有些数据传 keys"的寻址歧义（详见 `domain.md` §2 寻址决策）。

```yaml
# 每个数据项一条叶子 key；共享父系前缀即可整组查询。
- key: 船舶.海况.海水温度
  表: C
  量语义: 瞬时
  网格: "*/10 * * * *"
  单位: ℃
  规则: { kind: 有界瞬时, 基线: 2.5, 幅度: 1.0 }
- key: 船舶.海况.有义波高
  表: C
  量语义: 瞬时
  网格: "*/10 * * * *"
  单位: m
  规则: { kind: 有界瞬时, 基线: 3.5, 幅度: 1.5, 潜变量: 海况 }
# 请求 keys=["船舶.海况"] 时，接口按同表父系前缀动态展开到这两个叶子 key。
# 航迹是例外：经纬度/航速/航向强耦合、一次算出，保留为单 key 的复合 value（产出），不展开。
```

> **不声明 D**：当前无"按开放实例集 × 时间轴"的消费（`domain.md` §2.4）。将来若需，按已命名的 D 角扩展，resolver 增加"先 filter 选实例、再按 time 取序列"一步即可。

## 3. ★约束/派生层 `constraints.yaml`（跨 key，单 key schema 装不下）

> 物料守恒、产量上界都**牵涉多个 key**，塞不进任一 key 条目，故独立成层，由 resolver 求值时套用。

```yaml
# INV-1/2 物料守恒 + 得率派生（虾油线，挂在 C 累计上）
物料守恒.虾油线:
  基准: 工厂.虾油线.生产数据.虾粉消耗量          # 独立基准（累计）
  产出份额:                                    # Σ份额 ≤ 1，平滑噪声归一
    工厂.虾油线.生产数据.成品油产量: { kind: 份额, 基线: 0.18 }
    # 脱脂虾粉、回收酒精等按剩余份额派生
  得率: { key: 工厂.虾油线.生产数据.虾油得率, 定义: 成品油产量 / 虾粉消耗量 }   # INV-2 派生

# INV-1 捕捞量 ≥ Σ各线投料
物料守恒.捕捞:
  基准: 船舶.捕捞.累计产量
  下游: [ 船舶.虾粉线.流量秤.当天累计, 船舶.冻虾线.流量秤.当天累计, 船舶.虾肉线.流量秤.当天累计 ]
  约束: Σ下游 ≤ 基准

# INV-5 实际产量 ≤ 设计能力（上界 clamp）
上界.虾粉线: { 受约束: 船舶.虾粉线.打包.当天累计重量, 上界: 船舶.虾粉线.设计能力.日产量 }

# INV-4 温度链 + 中心温度 ≤ 冷库温度
温度链.冻虾:
  序: [捕捞≤加工≤冷链≤仓储≤检测]            # 时间因果
  约束: 检测#冻块中心温度 ≤ 船舶.冻虾舱.温度
```

> 实现策略（A 结构性派生优先于 B 钳制）见《造数引擎》§5。依赖图必须无环（INV-3；回收料按新批次入账、不回指上游）。

## 4. 溯源声明（B · value=`{类别, 溯源链条}`，数据归各环节、引用不复制）

> 以甲方 6.x 为权威：value 仅 `{类别, 溯源链条}` 两个顶层字段；**追溯/检验是链条内某环节的数据**（不拍平成顶层）。

```yaml
- key: 溯源
  表: B
  成熟度: 人工固定           # Fake 阶段配置喂；结构 = 回溯产出结构
  filter: [产品批号]
  环节顺序:                  # 按产品类别决定路径（路径长短不同）
    全脂虾粉: [磷虾捕捞, 船载加工, 冷链运输, 岸基存储, 产品检测]
    虾油:     [磷虾捕捞, 船载加工, 冷链运输, 岸基存储, 虾油提取, 产品检测]
    虾油胶囊: [磷虾捕捞, 船载加工, 冷链运输, 岸基存储, 虾油提取, 胶囊封装, 产品检测]
  条目:
    - filter: { 产品批号: 2606AKM01 }
      value:
        类别: 全脂虾粉溯源
        溯源链条:                       # 每个环节 = 批次编号 + 本环节数据（过程数据用引用）
          磷虾捕捞:                      # 捕捞环节自带捕捞信息（即三方接口拍平的"原料来源追溯"）
            批次编号: LXBL-2026-001
            捕捞海域: { 经度: 120.5, 纬度: 35.2 }
            捕捞方式: 拖网捕捞
            捕捞时间: "2026-06-15 08:00:00"
            原料溯源: null               # 可递归→上游批号（虾油/蛋白肽）
          船载加工:
            批次编号: CZJG-2026-001
            引用: [船舶.虾粉线.打包.当天累计重量, 船舶.虾粉线.蒸煮温度.直接蒸煮器]   # 不抄独立值
          冷链运输:  { 批次编号: LYYS-2026-001 }
          岸基存储:  { 批次编号: AJCC-2026-001 }
          产品检测:                      # 检测环节自带检验（即三方接口拍平的"产品检验"）
            批次编号: 2606AKM01
            引用: { key: 检测, filter: { 产品: 全脂虾粉, 批次: 2606AKM01 } }
```

> 与甲方 6.x 一致：追溯=捕捞环节数据、检验=检测环节数据，都在链条内；过程数据凭批次编号引用对应 key，**不在溯源里抄独立值**。

## 5. 取数解析契约（resolver 行为）

一次查询 `q = (key [, filter] [, time])`：

```
1. spec = registry[key]            # 无 → 404
2. 表校验：A 不收 filter/time；B 收 filter；C 收 time（点查=时刻，区间=起止）
3. override 命中(overrides.yaml)？ → 返回覆盖/冻结/兜底值
4. 按成熟度分派取值：
     人工固定 → 读 config 条目（B 按 filter 过滤）
     规则生成 → 造数引擎求值（C 按网格枚举）
     真值采集 → provider 取数（sqlite / 远程时序库 / RTSP 地址）
5. 套用 constraints.yaml 中涉及本 key 的派生/约束（求值期，同一 t 递归、记忆化）
6. 按表组装响应：A=值；B=命中条目（0~N）；C=时序（每点带 time 字段；点查单点 / 区间多点）
```

- **对外每表一个接口、各自批量**（实现层一个 resolver 统一，接口层按形状分——`domain.md` §8）：
  `/api/value`(A)、`/api/records`(B keys+filter)、`/api/series`(C keys+window)。不做单一 `/api/query`。
- ★**三端点同构：`keys` + 一个辅助参数**。辅助参数名**随轴而定**（不复用）：B 的轴是「选择」→ `filter`（`{key: {字段: 值}}`）；C 的轴是「时间」→ `window`。**时间是独立的轴，不是 filter 字段**（`domain.md` §2 核心判断），故 C 绝不叫 filter。
- **`/api/series` 的 `window` 支持两种形态**：全局形态 `{start?, end?, points?}` 应用到本次 `keys` 展开的全部 C key；逐 key 形态 `{key: {start?, end?, points?}}` 允许每个 key 独立配置。两种形态不能混用，混用 → 400。
- **`window` 缺省回退**：start/end 不给=当前时刻单点；`points` 不给→全局 20。
- **`window` 逐条目自洽校验**：逐 key 形态下条目 key 须在 `keys` 内，且为 C 表 key 或能展开到 C 表后代的父系前缀；`start/end` 成对且 `end≥start`；`points` 正整数。任一不满足 → 400（不静默放行）。
- **时间模式**（每个 window 条目内）：`start/end` 都不给=当前时刻单点；`start==end`=该时刻单点；`end>start`=区间。
- **寻址只用 `keys`**（无 `metrics`）：C 表只声明扁平叶子 key（§2.3）。三端点收到同表父系前缀时，会自动展开为其后代叶子 key 取数，响应形状等同于直接传入所有叶子 key；key 发现走 `/api/keys` 的叶子 key 列表。
- **`显示` 是可选展示名**：registry 条目可声明 `显示`；`/api/series` 与 `/api/keys` 透出该字段，供前端展示，稳定寻址仍以 key 为准。
- 派生(得率/百分比)由 constraints 的 `定义` 表达式在 resolver 求值；引用(如车间外大气→天气)解析对应 key、不复制。
- 协议字段（key/filter/window/start/end/points/value/status）英文；业务名与 value 内部键中文。
- 单位拼接、图表结构归前端。

### 5.1 C 表区间查询的固定点数 / 自适应比例尺（已实现）

前端按像素宽度消费曲线：查一天和查一年都只要 ~N 个点。故 C 表区间查询接收**点数**而非步长，由后端自适应比例尺：`step = (end − start) / N`，在 N 个桶上各产一个点，`step` 随跨度自动缩放，**一套逻辑覆盖所有比例尺**。

**N 的来源（逐 key 求）**：

```
N(key) = window.points             # ① 全局 window
      或 window[key].points        # ② 逐 key window
      ?? 20 (DEFAULT_POINTS)        # ③ 全局硬默认
      → clamp 到 [1, 2000(MAX_POINTS)]   # 上限兼内存护栏
```

> 设计取舍：**点"数量" N** 属视图/像素宽度，是请求方关注点，故仅由请求 `points` 覆盖；不再提供逐 key 配置默认，避免同一接口出现多套隐式点数规则。**聚合"方法"**则不另设配置——直接复用每 key 已声明的 `量语义` 分派（下表）。

每桶取值，**按已声明的 `量语义` 分派**：

| 量语义 | 每桶采样时刻 | 说明 |
|---|---|---|
| 瞬时 | 桶中心 | 够用；季节/趋势包络本就盖过噪声 |
| 累计 | 桶右端（`C(桶末)`，累计曲线） | O(1) 求值，精确；"每段产量柱"=`C(桶末)−C(桶头)` 为将来可选项 |
| 离散状态 | 桶中心（默认）/ 桶内众数 | — |

> **小跨度不降采样**：若 `[start,end]` 内网格点数 ≤ N，逐网格点返回（与历史行为一致，如查 6 小时整点温度仍恰 7 点）。采样时刻一律 `align` 到网格，落在确定性网格点上、与点查同值。

**跨缩放一致性**：值是 `f(key,t)` 纯函数，任何查询取同一 `t` 必逐字相等（确定性，永不破）。朴素 `range/N` 网格跨缩放不对齐——月视图与年视图的点落在不同时刻、几乎不精确重合，但二者只是同一曲线疏密不同的采样，**不会矛盾**，互相 refine。累计量「每段产量」在不同跨度下桶宽不同、数值天然不同——是**不同窗口的聚合，非不一致**；底层 `C(t)` 在相同时刻仍相等。

> 不做 LTTB 等保形降采样、不预建多分辨率汇总表——现算在任意时刻/任意间距免费求值（《造数引擎》§9.1）。遇到某瞬时量噪声远大于趋势、粗缩放显毛糙时，再把该类改桶内平均即可。
