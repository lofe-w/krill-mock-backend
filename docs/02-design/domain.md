## 0. 这套系统的定位（先把源头摆正）

本系统是"深蓝号"磷虾船 + 连云港工厂可视化大屏的**数据后端**，现阶段绝大多数数据为假数据，少量接真实源，将来整体迁移到真实后端。

它的价值**不在**"做一台多快、多通用的供数机器"，而在**一件事**：

> 用一个**最小的、与实现无关的、正确的模型**，把这块大屏需要的数据语义**一次性表达对**——使它现在能驱动大屏、将来能无痛迁移、且全程不在前端呈现错误、矛盾或将来实现不了的数据。

由此推出三条纪律，贯穿全文：

1. **按本质建模，对实现免疫**：模型只描述"数据本质是什么形状"，绝不按"值怎么造出来"（真值/规则/采集）组织。后者只是每个数据的一个旁注属性（§7），将来整体消失。
2. **模型是源头，实现与接口是下游**：造数引擎、对外接口都是本模型的可丢弃投影，排在最后（§8、§9）。
3. **与真实后端对齐**：模型里每个概念都应能和未来真实 MES/溯源系统的概念对上，这是"减少前端改造"的唯一兑现方式。

## 1. 业务全景：一条"物料不断被转化"的链

甲方清单描述的不是零散数据，而是**同一批磷虾物料沿产业链被一次次转化**的过程：

```
环境/海况 ─影响─▶ 捕捞 ─▶ 原料磷虾
                          ├─▶ 虾粉线 ─▶ 全脂虾粉
                          ├─▶ 冻虾线 ─▶ 冻虾          各自抽检
                          └─▶ 虾肉线 ─▶ 虾肉
   ─▶ 冷链运输 ─▶ 岸基仓储 ─▶ 陆地精加工
                          ├─ 虾油线：全脂虾粉 ─▶ 虾油 + 脱脂虾粉 + 回收酒精 ─▶（封装）虾油胶囊
                          └─ 蛋白肽线：脱脂虾粉 ─▶ 蛋白肽 + 渣粉
   ─▶ 产品溯源（7 类：全脂虾粉/冻虾/虾肉/虾油/虾油胶囊/脱脂虾粉/蛋白肽）
```

**本质命题**：整个企业是一张**物料转化网络**——节点是**物料批次**，边是**生产作业**（一次作业消耗输入批次、产出输出批次）。所有读数、产量、能耗、得率、检验都是挂在"某次作业/某个批次/某台设备"上的观测；**溯源是在这张网络上沿批次回溯得到的视图，不是一份独立数据**（详见 §5）。

这层业务理解是建模的"源头认知"。但本系统是**供数后端**，它真正要交付的，是把这些数据**以什么形状对外承载**——于是进入 §2。

## 2. 建模的本质：两根正交轴 → 四种数据形状

把大屏需要的每一个数据，抛开它是温度、产量还是溯源，只问两个**本质问题**：

- **轴一·要不要"选择"？** 它是单一的一个值，还是要从一个**实例集**里按身份挑一条？
- **轴二·有没有"时间"？** 它是站立不动的事实，还是**以时间为轴**演化的观测？

两轴正交，生成**四种数据形状**（这是本系统的核心模型）：

| | 无时间（站立事实） | 有时间（时序演化） |
|---|---|---|
| **无选择**（单一） | **A** `(key, value)` | **C** `(key, time, value)` |
| **有选择**（按身份挑） | **B** `(key, filter, value)` | **D** `(key, filter, time, value)` |

> 关键判断：**时间是一根独立的轴，不是 filter 里的一个字段。** "这个数据有没有历史"是本业务链最深的一条区分（温度会变、IMO 号不变），必须显式成轴。

### 2.1 key 与 filter 的分工判据（什么进 key、什么进 filter）

- **key**：编码**封闭、已知**的命名空间——实体/子实体/指标。这类集合可枚举，直接进 key（如 `船舶.冻虾舱.温度`、`工厂.虾油线.提取温度`）。
- **filter**：仅用于在**开放、数据驱动**的实例集里"选一条"——典型是**批号**（无法把所有批号枚举成 key）。
- 推论：`指标`、`设备`、`产线` 属封闭集 → 进 key；`产品批号` 属开放集 → 进 filter。这条判据决定了形状归属，避免靠手感。

### 2.2 value 的内部轴（黑箱里还有结构）

形状只管"外面怎么取"，`value` 内部还有一根独立的建模轴：

- **标量**：`"9849332"`、`-11.2`
- **记录**：一组字段的对象（仓储记录、检验报告）
- **引用**：value 中**指向其它 key**，而非复制数据（尤其溯源，见 §5）
- **递归**：value 含指向上游同类的引用（批次谱系）

### 2.3 时序的 value 语义：采样观测 vs 离散事件

C/D 两种时序形状，其 value 有两类本质不同的语义，**建模时必须区分**：

- **采样观测**：按时间网格取到的读数。又分**瞬时**（温度、航速）与**累计**（累计捕捞量；窗口量 = 末值 − 首值）。
- **离散事件**：逐次发生的业务事实，时间是**事件发生时刻**而非采样点。对应甲方 5.3「每一次生产的输入品/输出品」——一次生产作业的投入产出包。**若大屏需展示"每一次生产"，按事件建模，勿混入采样序列。**

## 3. 业务数据归属（合体表）

> 一级骨架 = 四形状；"怎么来的"见 §7，不进此表骨架。

| 业务数据 | 形状 | key 示例 | value 装什么 |
|---|---|---|---|
| 船舶信息（IMO/船名） | A | `船舶.IMO` | 标量/小记录 |
| 捕捞系统规格（铭牌） | A | `船舶.桁杆泵吸系统.规格` | 尺寸/设计能力记录 |
| 产线设计能力 | A | `船舶.虾粉线.设计能力` | {日产量, 年产量…}（是上界，非时序） |
| 工厂信息 | A | `工厂.信息` | {经纬度, 地址, 编号} |
| 渔季 | A | `渔季` | 区间列表（可跨年） |
| 溯源（7 类） | **B** | `溯源` · filter `产品批号` | **谱系投影**：环节链 + 原料追溯 + 检验（§5） |
| 产品检测 | B | `检测` · filter 批次/产品 | 检测项目 + 结论 + 机构 |
| 仓储/物流 | B | `仓储` · filter 批号/仓库类别 | 9 字段记录 |
| 设备台账 | B | `设备` · filter 所属产线 | 铭牌 + 所属工艺 |
| 专利/商品 | B | `专利` · filter 所属 | 记录 |
| 人员 | B | `人员` · filter 所属 | 档案 + **内嵌时序属性**（§4.2） |
| 捕捞产量（累计） | C | `船舶.捕捞.累计产量` | 累计值；窗口量 = 末 − 首 |
| 捕捞系统运行参数 | C | `船舶.拖网.钢丝绳受力` | 瞬时（航速/绞车/吸虾泵…） |
| 船载三线工艺/设备读数 | C | `船舶.虾粉线.蒸煮温度` | 瞬时/累计（流量秤/速冻温度/打包累计…） |
| 设备运行状态 | C | `船舶.虾肉线.速冻机.状态` | 离散枚举时序 |
| 船舶能耗 | C | `船舶.能耗.剩余燃油` | 时序 |
| 工厂两线工艺/能耗/环境 | C | `工厂.虾油线.提取温度` | 时序（MVR真空度/酶解PH/得率/车间温湿度…） |
| 船舶航行 | C | `船舶.航行.航速` | 时序（航迹 = 位置时序连线） |
| 天气/海况 | C | `船舶.海况.有义波高` | 时序（天气/风/流/海冰/叶绿素） |
| **某批次的冷链温度曲线** | **D** | `冷链温度` · filter `批号` | 该批次随时间的温度序列 |

> D 形状的存在价值：溯源里"冷链各段温湿度随时间"既要按**批号选**、又是**时序**——这正是 (选择 + 时间) 的第四个角，单靠 A/B/C 无法承载。

## 4. 两个关键判断（体现"按本质而非表象")

### 4.1 溯源、仓储是 B 而非 A
因为它们要在**开放实例集**（批号）里选一条。仓储若按"整体取全部"消费则退化为 A——**取决于大屏消费方式**，需与业务确认。

### 4.2 人员有时间，为什么是 B 不是 C
判据是**消费方式**，不是"数据里有没有时间"：
- 大屏**按时间切片**查某属性曲线 → 才是 C/D。
- 大屏**整体取人、把变更历史内嵌着看** → 是 B，时间作为 `value` 内部的**内嵌结构**，不是查询轴。

人员的职务/位置/在岗是 value 内嵌的时序属性（升序变更数组），整人随 B 返回。**同一份带时间的数据，既可做成 C（可查序列）、也可内嵌进 A/B 的 value——由消费方式决定。**

## 5. 溯源 = 谱系投影（不是写死的链条）

溯源**没有自己独立的数据**。它的 value 定义为：

> 从一个**产品批次**出发，沿**批次谱系**（哪批料产出哪批料，构成有向无环图）向上游回溯，在经过的**每一个环节**上，把该环节本就存在的事实（捕捞海域/时间、船载加工工艺与累计产量、冷链各段温湿度与转运时间、岸基仓储到库时间与温湿度、各级检验）**聚合**成有序视图。

由此：

- **7 类产品"像又不同"，只是谱系路径长短不同**：船上直接品（冻虾/虾肉/全脂虾粉）路径为 捕捞→船载加工→冷链→岸基仓储→检测；岸上精加工品（虾油/胶囊/脱脂虾粉/蛋白肽）在其后再接 提取/酶解→（胶囊再接封装）→工厂信息→成品检测。**路径由谱系自然决定，不需为 7 类各写一套静态链。**
- **value 用引用、不复制**：溯源各环节引用的冷链温度、加工参数等，本就是别处的 C/D 形状数据；溯源 value 应**引用其 key**，而非把数据抄进来。
- **原料追溯天然递归**：谱系是递归 DAG，回溯到上游批次后继续上溯即可。

> 现状：Fake 阶段可用配置喂溯源数据，但**配置的结构必须 = 回溯会产出的结构**（环节顺序由产品类别决定、原料递归指向上游批号），以保证将来切回"真实回溯"时前端零改造。**这是与真实后端对齐的关键点。**

## 6. 关键业务不变式（只取大屏看得见的）

为避免前端呈现自相矛盾的数据，下列不变式须在所产出的值之间成立。**判据**：观众盯着大屏有无可能当场抓到矛盾——只强制"看得见"的那些。

1. **物料守恒**：一次作业输出总量 ≤ 输入总量。捕捞量 ≥ 三线投料之和；1 批全脂虾粉 → 虾油+脱脂虾粉+回收酒精，重量和 ≈ 投入。（仅当大屏同屏可核对时强制，如 5.3。）
2. **得率是派生量**：得率 = 输出/投入 ∈ (0,1]，不可与输入输出各自独立取值。
3. **谱系无环可达**：任一成品批可沿谱系回溯到捕捞，且无环。
4. **冷链温度安全且单调时序**：中心温度 ≤ 冷库温度；环节时间满足 捕捞 ≤ 加工 ≤ 冷链 ≤ 仓储 ≤ 检测。
5. **设计能力是上界**：实际日/年产量 ≤ 铭牌设计能力。

> 不变式的**实现手段**（按构造派生、共享潜变量等）见《造数引擎》——那是实现，不是模型。

## 7. 数据成熟度（实现旁注，正交于形状，将来消失）

每个 key 另带一个**成熟度**属性，与形状**正交**，**绝不**作为模型骨架：

| 成熟度 | 含义 | 覆盖范围 | 接入方式 |
|---|---|---|---|
| 真值采集 | 接真实源 | 工厂天气；船舶航行/海况；监控视频 | 天气：采集落本地库直查；航行/海况：远程时序库只读透传；视频：RTSP 循环推流返回地址 |
| 规则生成 | 确定性造数 | 绝大多数船/厂读数、产量、状态、得率 | 见《造数引擎》（可复现纯函数） |
| 人工固定 | 配置给定 | 铭牌、批次基本信息、检验结论、人员名册、溯源结构 | 配置即数据 |

> "船舶冻虾舱温度（规则生成）"与"工厂天气温度（真值采集）"**本质同为 C 形状**，只是成熟度不同。模型对二者一视同仁——这正是"对实现免疫"。

## 8. 对外供给与接口（最上层投影，最可替换）

接口是四形状的**外部投影**，是整套设计**最上层、最不重要、最可替换**的一层，**待模型定稿后再细化**，现仅定原则：

- 一个统一查询入口，按数据的**形状**自然分派：A 直接给值；B 给（可多条）选中记录；C 给带 `time` 的序列（点查给时刻、区间给起止）；D 同 C 但附 `filter` 选实例。
- **统一发生在实现层**（一套分发机制服务四形状），**不在模型层**假装只有一种形状——避免把单一事实硬塞进"带空 filter 的数组"这类错位。
- 协议字段（key/filter/time/value/status）用英文；业务名与 value 内部键用中文。
- 不做鉴权/分页/admin/展示层；单位拼接、图表结构归前端（关注点分离）。

## 9. 实现要点（指向附录）

- **可复现**：规则生成的值 = 对 `(key, 网格时间)` 哈希播种的纯函数，历史任意时刻查询结果一致；逼真性用相干噪声 + 分形叠加；不变式用按构造派生。**完整算法见《造数引擎》。**
- 技术选型、落库范围、流媒体等属实现细节，随开发在代码与附录中沉淀，**不进本模型文档**。
