## 0. 这套系统的定位（先把源头摆正）

本系统是"深蓝号"磷虾船 + 连云港工厂可视化大屏的**数据后端**，现阶段绝大多数数据为假数据，少量接真实源，将来整体迁移到真实后端。

它的价值**不在**"做一台多快、多通用的供数机器"，而在**一件事**：

> 用一个**最小的、与实现无关的、正确的模型**，把这块大屏需要的数据语义**一次性表达对**——使它现在能驱动大屏、将来能无痛迁移、且全程不在前端呈现错误、矛盾或将来实现不了的数据。

三条纪律，贯穿全文：

1. **按本质建模，对实现免疫**：模型只描述"数据本质是什么形状"，绝不按"值怎么造出来"（真值/规则/采集）组织。后者只是每个数据的旁注属性（§7），将来整体消失。
2. **模型是源头，实现与接口是下游**：造数引擎、对外接口都是本模型的可丢弃投影，排在最后（§8、§9）。
3. **形状由消费决定，不由数据是否存在决定**：甲方清单告诉你"有哪些数据"，前端接口揭示"这些数据怎么被消费"。一个数据归哪个形状，取决于大屏怎么取它，而非它在现实中是什么。

## 1. 业务全景：一条"物料不断被转化"的链

```
环境/海况 ─影响─▶ 捕捞 ─▶ 原料磷虾
                          ├─▶ 虾粉线 ─▶ 全脂虾粉
                          ├─▶ 冻虾线 ─▶ 冻虾          各自抽检
                          └─▶ 虾肉线 ─▶ 虾肉
   ─▶ 冷链运输 ─▶ 岸基仓储 ─▶ 陆地精加工
                          ├─ 虾油线：全脂虾粉 ─▶ 虾油 + 脱脂虾粉 + 回收酒精 ─▶（封装）虾油胶囊
                          └─ 蛋白肽线：脱脂虾粉 ─▶ 蛋白肽 + 渣粉
   ─▶ 产品溯源（7 类：全脂虾粉/冻虾/虾肉/虾油/虾油胶囊/脱脂虾粉/蛋白肽）
```

**本质命题**：整个企业是一张**物料转化网络**——节点是**物料批次**，边是**生产作业**。所有读数、产量、能耗、得率、检验都是挂在"某次作业/某个批次/某台设备"上的观测；**溯源是沿批次回溯得到的视图，不是一份独立数据**（§5）。

这是建模的"源头认知"。但本系统是**供数后端**，真正要交付的是把这些数据**以什么形状对外承载**——进入 §2。

## 2. 建模的本质：两根正交轴 → 四个角，实例化为三张表

把大屏需要的每一个数据，抛开它是温度、产量还是溯源，只问两个**本质问题**：

- **轴一·要不要"选择"？** 单一的一个值，还是要从一个**实例集**里按身份挑一条？
- **轴二·有没有"时间"？** 站立不动的事实，还是**以时间为轴**演化的观测？

两轴正交 → **四个角**：

| | 无时间（站立） | 有时间（时序） |
|---|---|---|
| **无选择** | **A** `(key, value)` | **C** `(key, time, value)` |
| **有选择** | **B** `(key, filter, value)` | **D** `(key, filter, time, value)` |

> 关键判断：**时间是一根独立的轴，不是 filter 里的一个字段。** "这个数据有没有历史"是本业务链最深的一条区分（温度会变、IMO 号不变），必须显式成轴。

**实例化只用三张表（A/B/C）。** 据前端消费方式查证（§2.4 论据）：当前大屏**没有任何数据**需要"按开放实例集选 + 时间轴"同时成立，故 **D 的第四个角当前无人消费，记为潜在扩展（YAGNI）**，不实例化。两轴模型仍保留 D 这个名分——将来若出现"按批号看某批货冷链温度曲线"之类消费，D 是已命名、即插即用的扩展位。

### 2.1 key 与 filter 的分工判据
- **key**：编码**封闭、已知**的命名空间（实体/子实体/指标），可枚举（如 `船舶.冻虾舱.温度`、`工厂.虾油线.提取温度`）。
- **filter**：仅用于在**开放、数据驱动**的实例集里"选一条"——典型是**批号 / 仓库类别 / 商品编号**（无法枚举成 key）。

### 2.2 value 的内部轴
形状只管"外面怎么取"，`value` 内部还有一根建模轴：
- **标量** `"9849332"`、`-11.2`
- **记录** 一组字段（仓储记录、检验报告）
- **窗口聚合** 累计量按 {当天/当月/当前渔季/年度} 端点求差（甲方/前端均以此消费产量）
- **枚举维度（字典）** 成分项（磷脂/EPA/DHA/虾青素/酸价…）是跨产品复用的封闭枚举，作为 value 内的子键，对齐未来 MES 质检项字典
- **引用** value 中**指向其它 key 或批次编号**，而非复制数据（尤其溯源，见 §5）

### 2.3 时序的 value 语义：采样观测 vs 累计
C 表的 value 有两类：
- **瞬时**：温度、航速、受力、车间环境…
- **累计**：捕捞量、产量、各类消耗量、耗电/耗油…；窗口量 = 末值 − 首值。前端正是以 {当天/当月/当前渔季/年度} 窗口消费它（见前端接口 10.x、4.9/4.10）。

> 注：甲方 5.3「每一次生产的输入/输出」在现实中是离散事件，但**前端不按"每一次"消费**，而是消费成窗口累计的消耗量/产量 + 派生得率（前端接口里没有"逐次生产"端点）。故 5.3 归 **C（累计）+ 派生**，不引入事件形状。若将来确需展示"逐次生产"，再按 D（事件语义）扩展。

### 2.4 实例化：三张表（表视图）+ 论据

把四形状落成机器里的存储/容器，就是**三张通用表**（多个业务实体共表，靠 key 区分）：

| 表 | 列（字段） | 装哪些业务实体 | 为什么 |
|---|---|---|---|
| **A 表** `key · value` | key, value | 船舶信息、捕捞系统规格、产线设计能力、工厂信息、渔季 | 单一站立事实，取出即用，无需选择、无时间 |
| **B 表** `key · filter · value` | key, filter, value | 溯源、产品检测、仓储/物流、设备台账、人员、专利、商品、监控清单 | 站立事实，但要在开放实例集（批号/仓库类别/商品编号/所属）里选一条 |
| **C 表** `key · time · value` | key, time, value | 捕捞产量、三线工艺/设备读数、设备状态、能耗、工厂两线工艺/能耗/环境、航行、天气海况、各舱冷库温度、各类消耗/产量累计 | 以时间为轴演化的观测；瞬时或累计 |

> **为什么是三张而非四张**：D 的第四个角（选择+时间）当前无消费（§2.4 论据），不建。
>
> **D 不实例化的查证论据**（来自前端《API 文档》——揭示真实消费方式）：
> 1. **溯源接口**只返回每个环节的**批次编号** + 原料追溯 + 检验，**不返回任何"按批号的时间序列"** → 溯源是纯 B，零时间维。
> 2. **冷库/冷链温度**确有"温度趋势"，但全部**按舱/按车间**（冻虾舱温度趋势、虾粉舱温度趋势…）→ 是 C，从不按批号。
> 3. **生产消耗/产出**（甲方 5.3/5.4/5.5）被消费为**窗口累计**（当天/当月…）+ 得率 → C，不是按作业批次的时序。
> 4. 通查六个接口：无任何一处需要"按开放实例集选 + 时间轴"同时成立。
>
> 注：前端文档标注"初版/不稳定/仅供参考"，故 D 不删除其概念名分，只是当前不实例化。

## 4. 两个关键判断（体现"按本质而非表象"）

### 4.1 溯源、仓储、物流是 B 而非 A
因为它们要在**开放实例集**（商品编号/仓库类别）里选一条。（前端接口正是输入单个编号、返回一条。）

### 4.2 人员有时间，为什么是 B 不是 C
判据是**消费方式**，不是"数据里有没有时间"：
- 大屏**按时间切片**查某属性曲线 → 才是 C。
- 大屏**整体取人、把变更历史内嵌着看** → 是 B，时间作为 `value` 内部的内嵌结构，不是查询轴。

前端接口里人员就是一个列表，每人的职务/位置/在岗是 value 内嵌的升序变更数组，整人随 B 返回。**同一份带时间的数据，既可做成 C、也可内嵌进 A/B 的 value——由消费方式决定。**

## 5. 溯源 = 谱系投影，value = `{类别, 溯源链条}`

**以甲方文档为权威**（甲方 6.x）：溯源是一条**环节链**，每个环节自带本环节数据——

- 6.x.1 **磷虾捕捞**环节 → 捕捞海域/方式/时间（即三方接口拍平出去的"原料来源追溯"，本质是捕捞环节的数据）；
- 6.x.2 **船载加工** / 6.x.3 **冷链运输** / 6.x.4 **岸基仓储** → 各自的过程数据；
- 6.x.5 **产品检测**环节 → 检验报告（即三方接口拍平出去的"产品检验"，本质是检测环节的数据）。

> 故 value = **`{类别, 溯源链条}`**，仅两个顶层字段。`溯源链条` = 按类别顺序的环节，**每个环节 = {批次编号 + 本环节数据}**。
> 三方接口把"原料来源追溯""产品检验"列为与溯源链条平级的顶层字段，是它的重构——**不跟**；二者归位到捕捞环节、检测环节内。

由此：

- **引用不复制**：环节里的过程数据（船载加工产量/温度、冷链温湿度、产品检测报告）凭批次编号**引用对应 key**（`检测`、各 C 读数），不抄独立值——否则与 C 表同屏打架（违反 §6）。捕捞海域/方式/时间是捕捞批次的固有属性，随该环节人工固定即可。
- **7 类按谱系路径长短自然区分**：路径由"产品类别 → 环节序列"决定（甲方/前端附录 A），**不为 7 类各写一套静态链**；环节顺序由类别配置。
- **原料追溯天然递归**：虾油/蛋白肽的捕捞环节可递归嵌套其原料虾粉的溯源（按批号）。

> 现状：Fake 阶段配置喂溯源，但**配置结构 = 回溯产出结构**（`{类别, 溯源链条}`，环节自带数据、引用不复制），保证将来切真实回溯时前端零改造。

## 6. 关键业务不变式（只取大屏看得见的）

为避免前端呈现自相矛盾的数据，下列不变式须在所产出的值之间成立。**判据**：观众盯着大屏有无可能当场抓到矛盾——只强制"看得见"的那些。

1. **物料守恒**：捕捞量 ≥ 三线投料之和；虾油线消耗 ≈ 产出（全脂虾粉 ≈ 虾油+脱脂虾粉+回收酒精）。前端以窗口累计同屏展示这些量，故须自洽。
2. **得率是派生量**：得率 = 输出/投入 ∈ (0,1]，不可与消耗/产出各自独立取值。
3. **谱系无环可达**：任一成品批可沿谱系回溯到捕捞，且无环——**回收无水酒精按新批次入账、不回指上游**（防环）。
4. **冷链/温度安全**：冻品中心温度 ≤ 冷库温度；溯源各环节时间满足 捕捞 ≤ 加工 ≤ 冷链 ≤ 仓储 ≤ 检测。
5. **设计能力是上界**：实际日/年产量 ≤ 铭牌设计能力。

> 不变式的**实现手段**（按构造派生、共享潜变量等）见《造数引擎》——那是实现，不是模型。

## 7. 数据成熟度（实现旁注，正交于形状，将来消失）

每个 key 另带一个**成熟度**属性，与形状（表）**正交**，**绝不**作为骨架：

| 成熟度 | 含义 | 覆盖 | 接入 |
|---|---|---|---|
| 真值采集 | 接真实源 | 工厂天气；船舶航行/海况；监控视频 | 天气=采集落本地直查；航行海况=远程时序库透传；视频=RTSP 推流返地址 |
| 规则生成 | 确定性造数 | 绝大多数船/厂读数、产量、状态、得率 | 见《造数引擎》 |
| 人工固定 | 配置给定 | 铭牌、批次基本信息、检验结论、人员名册、溯源结构 | 配置即数据 |

> "冻虾舱温度（规则生成）"与"工厂天气温度（真值采集）"**本质同为 C**，只是成熟度不同。模型对二者一视同仁——这正是"对实现免疫"。

## 8. 对外供给与接口（最上层投影，最可替换）

接口是三表的**外部投影**，是整套设计**最上层、最可替换**的一层，**待模型定稿后再细化**，现仅定原则：

- 一个统一查询入口，按数据的**表（形状）**自然分派：A 直接给值；B 给（可多条）选中记录；C 给带 `time` 的序列（点查给时刻、区间给起止）。
- **统一发生在实现层**（一套分发机制服务三表），**不在模型层**假装只有一种形状。
- 协议字段（key/filter/time/value/status）用英文；业务名与 value 内部键用中文。
- 不做鉴权/分页/admin/展示层；单位拼接、图表结构归前端。

## 9. 实现要点（指向附录）

- **可复现**：规则生成的值 = 对 `(key, 网格时间)` 哈希播种的纯函数，历史任意时刻查询一致；逼真性用相干噪声 + 分形叠加；不变式用按构造派生。**完整算法见《造数引擎》。**
- 技术选型、落库范围、流媒体等属实现细节，随开发在代码与附录中沉淀，**不进本模型文档**。

| 各 C 指标网格频率 | 预设占位 | 真实 cron 待业务/演示需要确定 |
| 字段级清单 | 随开发补全 | 各 key 的指标/字段进配置 |
