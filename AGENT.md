## 1. 这个项目是什么

**磷虾船（深蓝号）+ 连云港工厂 可视化大屏的数据后端（Mock 为主）。**

为前端可视化大屏提供数据。绝大多数指标是**确定性造数**（假数据，规则即时计算、不落库、可复现）；少数指标接**真实数据源**（工厂天气来自 open-meteo、船舶航行/天气来自远程 TimescaleDB、监控视频走 RTSP）。对外只有三个通用接口：`series`（时序数值）、`records`（结构化记录，含溯源/物流/人员/设备…）、`surveillance`（监控视频清单）。

设计目标：**简单、快、通用、可配置、可复现**——一份配置驱动造数与供给，增删改指标/接口不动框架；同一时刻的数据由纯函数确定，历史任意时间查询结果一致。

技术栈：Python 3.11+ / FastAPI；computed 指标无状态、无数据库；真实数据接 SQLite / TimescaleDB / RTSP(MediaMTX)。

## 4. 关键决策记录（ADR 速查）

> 以下均为与甲方/前端/架构师讨论后**已拍板**的决策。接手者除非有充分理由，不要推翻重议。

1. **假数据策略 = 懒计算 / 规则即真相**：computed 指标用确定性纯函数即时计算，不落库、无常驻造数进程。理由：可复现、历史一致性免费、无中间状态漂移。
2. **无数据库 / 无状态（仅限 computed）**：只有真实数据才落地——工厂天气落本地 SQLite，船舶数据走远程 TimescaleDB；computed 全程无状态。
3. **三个接口，按"查询形态"划分**：`series`（按时间取数值序列）、`records`（按标识/条件取结构化记录，含溯源/物流/人员/设备/专利/监控）、`surveillance`（监控 RTSP 清单）。**刻意不叫 fact/dim**——因为溯源在业务上是"事实"，只是本系统中静态，故按"记录"这一检索形态归类。
4. **meta 接口暂不提供**：`unit`、`type` 随每条 `series` 响应自带（自描述）；渔季边界现阶段由前端持有。将来需要再加回，不破坏契约。
5. **聚合口径吸收进"指标定义"**：累计型指标用闭式 `C(t)`，窗口聚合 = 末值−首值（前端相减）；`series` 入参只需 `keys + start + end`，无聚合/多窗口/格式参数。
6. **命名约定**：协议/结构字段用英文（status/data/keys/filter/start/end/values/time/value/unit/type）；业务名（指标名、类型名、业务字段）用中文。时序点统一 `{time, value}`。
7. **真实数据源**：工厂天气 = 定时拉 open-meteo 真实数据写 SQLite、接口直查 SQLite；船舶航行/天气 = 透传远程 TimescaleDB（外部进程每 10min 写，本地不落库）；监控 = RTSP（MediaMTX 循环推片源），地址由 `records` 维护并指向本系统。
8. **source provider 抽象**：每个指标声明 `source.kind` ∈ `computed / sqlite / timescaledb`；取数 = `override → provider`，新增数据源不动接口与前端。
9. **造数引擎**：种子化纯函数（哈希播种）保证可复现；相干噪声 + fBm 保证平滑/缓慢/有波动性。**指标间硬约束**（如"冻虾舱温度 ≤ 海水温度""累计捕捞量 ≥ Σ原料流量"）用**指标依赖图（DAG）+ 同一 t 按构造派生**实现（无环、记忆化）。
10. **DDD 限界上下文**：指标供给、造数、溯源、记录（主数据）、真实数据接入、监控视频、配置（共享内核）。系统读重写轻，近似 CQRS 读模型。
11. **暂不做**：admin/控制接口、前端兼容的"组合模板层"、后端展示层、鉴权/分页。前端默认接受通用接口。
12. **编号规则**：预设占位，待业务方提供真实规则后调整。
