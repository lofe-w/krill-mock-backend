# 前端使用手册 · 数据取数 API

> 面向前端/接入方。本手册讲**怎么调接口拿数据**；`registry/*.yaml` 讲**每个 key 有哪些数据**。
> 两者合起来构成前端取数的完整知识库：**先在 yaml（或本手册末尾的「全量 key 目录」）里查到 key，再照本手册用对应接口请求。**

---

## 0. 一句话心智模型

系统对外只有**一种寻址：key**（点分全限定名，如 `船舶.海况.海水温度`）。每个 key 属于三张表之一，**表决定你用哪个接口、请求体长什么样、响应形状是什么**：

| 表 | 含义 | 接口 | 典型数据 |
|----|------|------|----------|
| **A** | 单一站立值（人工固定） | `POST /api/value` | 信息、规格、设计能力 |
| **B** | 台账/列表（可按字段过滤） | `POST /api/records` | 仓储、人员、设备台账、溯源、检测 |
| **C** | 时序（按时间轴取点/取区间） | `POST /api/series` | 海况、能耗、工艺读数、产量、库存 |

> **表和「成熟度」正交**：成熟度（人工固定/规则生成/真值采集）只决定值从哪来，**不影响你怎么调**。当前阶段全部为 Fake（C=规则生成、A/B=人工固定），前端无需关心。

三个接口**同构**：请求体都是 `{ "keys": [...] , <按 key 作用域的参数> }`，一次可批量查多个 key；响应都是 `{ "status": 200, "data": { "<key>": ... } }`，按 key 回填。

## 1. 接入准备

### 1.1 Base URL
本地：`http://localhost:8000`。联调环境以运维给出的地址为准。

### 1.2 鉴权（Bearer Token）
除 `/api/health` 外所有接口都要带 header：

```
Authorization: Bearer <token>
```

- **前端用「使用侧 Token」**（`API_TOKEN_APP`）：可调 `/api/value`、`/api/records`、`/api/series`。
- 「运维侧 Token」（`API_TOKEN_OPS`）权限更高，用于 `/api/keys`、`/api/reload`，**前端一般不需要**。
- Token 无效或缺失 → `401`，中文提示「缺少或无效的使用侧 Token」。
- 若后端 `AUTH_ENABLED=false`（部分本地环境），可不带 Token。

### 1.3 跨域（CORS）
浏览器直连需后端放行来源（`CORS_ORIGINS`）。若遇 CORS 报错，联系运维把你的域名加入白名单，或改用后端/网关代理转发。

### 1.4 在线调试
后端自带 Swagger：浏览器打开 `http://localhost:8000/docs`，点右上角 **Authorize 🔓** 填一次 Token，即可直接试跑（每个接口的示例就是一条可直接发送的真请求）。

## 2. `/api/value` —— A 表（单一站立值）

**请求**

```bash
curl -X POST http://localhost:8000/api/value \
  -H 'Authorization: Bearer <API_TOKEN_APP>' \
  -H 'Content-Type: application/json' \
  -d '{ "keys": ["船舶.信息", "工厂.虾油提取生产线.设计能力"] }'
```

**响应**：`data[key]` 直接是该 key 的值（对象或标量）。

```json
{
  "status": 200,
  "data": {
    "船舶.信息": { "IMO": "9849332", "船名": "深蓝号", "船籍": "中国", "类型": "南极磷虾远洋捕捞加工船" },
    "工厂.虾油提取生产线.设计能力": { "全脂虾粉日处理量": "...", "典型日产量": "...", "单位": "..." }
  }
}
```

> A 表值的字段由 yaml 决定，前端按需取字段。字段清单见「全量 key 目录 · A 表」。

---

## 3. `/api/records` —— B 表（台账/列表，可过滤）

B 表一个 key 下有多条记录。`filter` 可选，**必须按 key 作用域**，形如 `{ "<key>": { 字段: 值 } }`。

**请求（带过滤）**

```bash
curl -X POST http://localhost:8000/api/records \
  -H 'Authorization: Bearer <API_TOKEN_APP>' \
  -H 'Content-Type: application/json' \
  -d '{
        "keys": ["溯源"],
        "filter": { "溯源": { "产品批号": "2606AKO01" } }
      }'
```

**响应**：`data[key]` 是记录数组，每条 `{ "filter": {...}, "value": {...} }`。

```json
{
  "status": 200,
  "data": {
    "溯源": [
      { "filter": { "产品批号": "2606AKO01" }, "value": { "……环节数据快照……" } }
    ]
  }
}
```

要点：

- **不传 `filter` 或该 key 无匹配 → 返回该 key 全部条目**（前端可自行前端过滤）。
- 每个 key 支持哪些过滤字段见「全量 key 目录 · B 表」的 `filter` 列（如 `溯源` 按 `产品批号`，`仓储` 按 `仓库类别 / 产品批号`，`检测` 按 `产品 / 批次`）。
- `filter` 顶层项必须是**已注册的 B 表 key**，且值为对象，否则 `400`（避免无作用域的过滤被静默套到所有 key）。

## 4. `/api/series` —— C 表（时序，重点）

C 表是随时间变化的量。请求体：`keys` + 可选 `window`（按 key 作用域的时间窗）。

```jsonc
{
  "keys": ["船舶.海况.海水温度", "船舶.能耗.累计耗油量"],
  "window": {
    "船舶.海况.海水温度":  { "start": "2026-01-01 00:00:00", "end": "2026-12-31 00:00:00", "points": 12 },
    "船舶.能耗.累计耗油量": { "start": "2026-01-01 00:00:00", "end": "2026-06-25 00:00:00" }
  }
}
```

**响应**：`data[key]` = `{ "单位": ..., "values": [ { "time": "...", "value": ... }, ... ] }`。

```json
{
  "status": 200,
  "data": {
    "船舶.海况.海水温度": {
      "单位": "℃",
      "values": [
        { "time": "2026-01-01 00:00:00", "value": 1.8 },
        { "time": "2026-02-01 00:00:00", "value": 2.1 }
      ]
    }
  }
}
```

### 4.1 时间窗的三种模式（由 `start`/`end` 决定，无独立 time 参数）

| 你传的 | 含义 | 返回 |
|--------|------|------|
| `window` 里该 key 缺省，或不传 `start`/`end` | **当前时刻** | 单点（对齐到网格的最近时刻） |
| `start == end` | 该时刻 | 单点 |
| `end > start` | 区间 | 在 `[start, end]` 上**重采样为 N 点** |

- `points`（N）：区间重采样点数。**缺省回退顺序**：请求 `points` → 该 key 配置的默认点数 → 全局默认 `20`。上限 `2000`（内存护栏）。
- 时间格式统一 `"YYYY-MM-DD HH:mm:ss"`；`start`/`end` 成对出现。
- **时区**：所有时间按**业务本地时区**理解（默认 `Asia/Shanghai`，北京时间 UTC+8），不是 UTC。
- **网格对齐**：返回的 `time` 会对齐到该 key 的采样网格（如 `*/10 * * * *` = 每 10 分钟）。网格由 yaml 的 `网格` 字段决定，前端只读结果、无需自己算。
- **量语义**（瞬时/累计）影响区间内每桶取样时刻：瞬时取桶代表时刻的值，累计取到该时刻的累计量。前端照 `values` 画图即可。

### 4.2 分组容器 vs 叶子 key（**最容易踩的坑**）

C 表里，一个「父 key + 多指标」会在加载期展开成**独立子 key**，父 key 降为**分组容器**（只用于发现子 key，**本身不是时序、查它拿不到 values**）。

- 查**叶子 key**（如 `船舶.虾粉生产线.流量秤.实时读数`）→ 正常返回 `values`。
- 查**分组容器**（如 `船舶.虾粉生产线.流量秤`）→ 返回子 key 列表，供你发现：

```json
{
  "status": 200,
  "data": {
    "船舶.虾粉生产线.流量秤": {
      "status": 200,
      "key": "船舶.虾粉生产线.流量秤",
      "表": "C",
      "分组": true,
      "子": ["船舶.虾粉生产线.流量秤.实时读数", "船舶.虾粉生产线.流量秤.当天累计"],
      "note": "分组 key（容器），请查询其下具体子 key"
    }
  }
}
```

**前端正确姿势**：拿到 `分组:true` 的响应后，用返回的 `子` 列表里的具体 key 再发一次 `/api/series` 取值。哪些是分组容器、各有哪些子 key，见「全量 key 目录 · C 表 · 分组容器」。

### 4.3 派生值 / 占位 / 真值采集

- **派生**（如各线得率 `虾粉得率`）：响应会多一个 `"派生": "<表达式>"` 字段，`values` 是算好的值，前端照常用。
- **占位/说明项**：极少数未配规则的 key，`data[key]` 返回 `{ "note": "占位/说明项，未配置规则..." }`，没有 `values`。
- **真值采集待对接**（yaml 标 `目标成熟度: 真值采集`，如 `船舶.航行` 等）：当前用 fallback 规则出数，响应带 `"note": "真值采集·当前用 fallback 规则（待真实源就绪切换）"`。**接口和字段将来不变**，前端可照现在写。

## 5. 错误与排查

| 现象 | 含义 | 处理 |
|------|------|------|
| `data[key] = { "error": "key 未注册" }` | 该 key 不在（内存）注册表 | 先查拼写/是否在 yaml；**若 yaml 里明明有**，多半是后端注册表未热加载 —— 让运维调 `POST /api/reload` 或重启后端 |
| `data[key] = { "error": "X 属 Y 表，请用对应接口（A→/value, B→/records, C→/series）" }` | 用错了接口 | 按提示换接口 |
| `400` filter/window 形状错 | `filter`/`window` 没按 key 作用域 | 改成 `{ "<key>": {...} }` 形状 |
| `401` | Token 缺失/无效 | 检查 `Authorization: Bearer <token>` |
| CORS 报错 | 来源未放行 | 联系运维加白名单或走代理 |

> **「key 未注册」但 yaml 里有** 是最常见的困惑：注册表在后端启动时一次性载入内存（`REG = load()`），新增/改动 yaml 后需要 `/api/reload` 或重启才生效。

## 6. 运维/自检接口（前端了解即可）

- `GET /api/health`（**公开**，无需 Token）：存活探测，返回 key 总数、按表/成熟度分布、自检是否通过。可用作前端启动前的连通性检查。
- `GET /api/keys`（运维 Token）：列出全部 key 及其 `表 / 接口 / 成熟度 / 分组`。**这是「全量 key 目录」的实时来源**——本手册末尾的目录是某次导出的快照，以 `/api/keys` 和 `registry/*.yaml` 为准。
- `POST /api/reload`（运维 Token）：重新加载注册表（改完 yaml 后使新 key 生效，无需重启）。

---

## 7. 三步上手（Checklist）

1. **查 key**：在 `registry/*.yaml` 或末尾目录里找到目标 key，记下它的**表**。
2. **选接口**：A→`/api/value`，B→`/api/records`，C→`/api/series`；带上 `Authorization: Bearer <使用侧Token>`。
3. **取值**：从 `data["<key>"]` 读结果。C 表若拿到 `分组:true`，改用返回的 `子` key 再查一次。
