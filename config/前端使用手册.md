# 前端使用手册 · 数据取数 API

> 面向前端/接入方。本手册讲**怎么调接口、每个参数什么含义**；**每个 key 有哪些数据、字段是什么，直接查 `registry/*.yaml`**（或运维侧 `GET /api/keys`）。
> 用法：先在 yaml 里查到目标 key 和它的**表(A/B/C)**，再照本手册用对应接口和参数请求。
> 适用版本：`v1.0.3`（从 `v1.0.2` 升级时请先看 `config/变更记录.md` 的 1.0.3 迁移指引）。

---

## 0. 一句话心智模型

系统对外只有**一种寻址：key**（点分全限定名，如 `船舶.海况.海水温度`）。每个 key 属于三张表之一，**表决定你用哪个接口、请求体长什么样、响应形状是什么**：

| 表 | 含义 | 接口 | 典型数据 |
|----|------|------|----------|
| **A** | 单一站立值（人工固定） | `POST /api/value` | 信息、规格、设计能力 |
| **B** | 台账/列表（可按字段过滤） | `POST /api/records` | 仓储、人员、设备台账、溯源、检测 |
| **C** | 时序（按时间轴取点/取区间） | `POST /api/series` | 海况、能耗、工艺读数、产量、库存 |

> **表和「成熟度」正交**：成熟度（人工固定/规则生成/真值采集）只决定值从哪来，**不影响你怎么调**。当前阶段全部为 Fake（C=规则生成、A/B=人工固定），前端无需关心。

三个接口**同构**：请求体都是 `{ "keys": [...] , <按 key 作用域的参数> }`，一次可批量查多个 key；响应都是 `{ "status": 200, "data": { "<key>": ... } }`，按 key 回填。

如果请求命中了兼容 alias 或已废弃 key，响应顶层会额外带 `warnings`。业务数据仍只从 `data` 取；`warnings` 只用于迁移提示。当前 `v1.0.3` 正式 registry 暂未配置真实 deprecated key，但接口能力已经就绪。

`key` 对前端是业务寻址名，允许跨表复用；真正唯一身份是 `表 + key`。因为三个接口已经按表拆开，前端仍按当前方式调用即可：同一个 `key` 放到 `/api/value`、`/api/records`、`/api/series` 会分别查 A/B/C 表。

## 1. 请求参数总览（keys / filter / window / points）

三个接口共享同一套请求模型：**一个 `keys` 数组 + 一个辅助参数**。辅助参数因表而异——A 无、B 用 `filter`、C 用 `window`；其中 `filter` 固定按 key 作用域，`window` 支持全局或按 key 作用域。

### 1.1 `keys`（三接口通用，必填）

字符串数组，元素是全限定 key。**一次可传多个 key，响应按 key 回填到 `data["<key>"]`**，互不影响。

- key 具体有哪些、属于哪张表 —— 查 `registry/*.yaml`（每条声明的 `key:` 与 `表:`）。
- 同一次调用的 keys **必须同属一张表**（用该表对应的接口）；查错表会在该 key 上返回「X 属 Y 表，请用对应接口」。跨表请分开调用。
- 可以传同表父系前缀，接口会把结果展开到该前缀下全部同表叶子 key。父系前缀不需要在 yaml 中单独注册。
- 如果前端需要把 `/api/keys` 的结果作为列表主键/缓存主键，请使用 `qualified_key`，不要只用 `key`。

### 1.2 `filter`（仅 `/api/records`，可选）

B 表一个 key 下有多条记录，`filter` 用来筛选。

- **形状：按 key 作用域** `{ "<key>": { 字段: 值 } }`——每个要过滤的 key 各给一套条件。
- **支持哪些过滤字段**：查 yaml 里该 key 的 `filter:` 声明（例：`溯源`→按 `产品批号`；`仓储`→按 `仓库类别`/`产品批号`；`检测`→按 `产品`/`批次`；`设备.台账`→按 `所属产线`/`设备名称`）。
- 一个 key 内多字段是 **AND**（全部命中才返回该条）。
- **不传 `filter`、或某 key 无匹配 → 返回该 key 的全部条目**（前端可再自行过滤）。
- 顶层项必须是**已注册的 B 表 key** 且值为对象，否则 `400`（防止无作用域的条件被静默套到所有 key）。

### 1.3 `window`（仅 `/api/series`，可选）

C 表的轴是时间，`window` 给 key 指定时间窗。

- **全局形态** `{ "start", "end", "points" }`：同一个时间窗应用到本次 `keys` 展开的全部 C 表 key。
- **逐 key 形态** `{ "<key>": { "start", "end", "points" } }`：每个 key 单独配置时间窗；key 可以是 C 表叶子 key 或能展开到 C 表的父系前缀。
- 两种形态不能混用；三者皆可选、逐条目自洽。
- `start` / `end`：区间端点，支持 `"YYYY-MM-DD HH:mm:ss"` 或 `"YYYY-MM-DD"`，**成对出现**。日期-only 写法中，`start` 按当天 `00:00:00`，`end` 按当天 `23:59:59` 理解。取值模式见 §4.1。
- 不在 `window` 里的 key，或不给 `start`/`end` → 取**当前时刻单点**。

### 1.4 `points`（`window` 内，可选）

区间重采样的点数 N。缺省为全局默认 `20`；上限 `2000`（内存护栏）。仅在区间模式（`end > start`）下有意义。

### 1.5 批量与联动（**一次调用查多个 key**）

三接口都是「`keys` 批量 + 按 key 作用域的辅助参数」，所以**一次请求就能取多个 key 的数据，且每个 key 带自己的过滤/时间窗，互不干扰**：

- **B 表**：`keys` 列多个台账/清单 key，`filter` 里对每个 key 各给筛选条件，一次拿回多份过滤结果。
- **C 表**：`keys` 列多个时序 key，`window` 可给一个全局时间窗，也可对每个 key 各给时间窗和点数，一次拿回多条曲线。
- 逐 key 辅助参数只对**它点名的那个 key**生效；没点名的 key 走默认（B=全量、C=当前时刻单点）。全局 `window` 则作用于本次 `keys` 展开的全部 C 表 key。
- 响应统一 `data["<key>"]` 回填，前端按 key 取用即可。

> 例：`{ "keys": ["溯源","检测"], "filter": { "溯源": {"产品批号":"2606AKO01"}, "检测": {"产品":"成品虾油"} } }` —— 一次调用，溯源按批号、检测按产品，各自过滤。

## 2. 接入准备

### 2.1 Base URL
本地：`http://localhost:6632`。联调环境以运维给出的地址为准。

### 2.2 鉴权（Bearer Token）
除 `/api/health` 外所有接口都要带 header：

```
Authorization: Bearer <token>
```

- **前端用「使用侧 Token」**（`API_TOKEN_APP`）：可调 `/api/value`、`/api/records`、`/api/series`。
- 「运维侧 Token」（`API_TOKEN_OPS`）权限更高，用于 `/api/keys`、`/api/reload`，**前端一般不需要**。
- Token 无效或缺失 → `401`，中文提示「缺少或无效的使用侧 Token」。
- 若后端 `AUTH_ENABLED=false`（部分本地环境），可不带 Token。

### 2.3 跨域（CORS）
浏览器直连需后端放行来源（`CORS_ORIGINS`）。若遇 CORS 报错，联系运维把你的域名加入白名单，或改用后端/网关代理转发。

### 2.4 在线调试
后端自带 Swagger：浏览器打开 `http://localhost:6632/docs`，点右上角 **Authorize 🔓** 填一次 Token，即可直接试跑（每个接口的示例就是一条可直接发送的真请求）。

## 3. `/api/value` —— A 表（单一站立值）

参数：仅 `keys`（无辅助参数）。响应 `data[key]` 直接是该 key 的值（对象或标量，字段见 yaml）。

```bash
curl -X POST http://localhost:6632/api/value \
  -H 'Authorization: Bearer <API_TOKEN_APP>' \
  -H 'Content-Type: application/json' \
  -d '{ "keys": ["船舶.信息", "工厂.虾油提取生产线.设计能力"] }'
```

```json
{
  "status": 200,
  "data": {
    "船舶.信息": { "……该 key 的值对象，字段见 yaml…… " },
    "工厂.虾油提取生产线.设计能力": { "……" }
  }
}
```

传 A 表父系前缀时，响应会直接展开为多个 A 表叶子 key：

```json
{
  "keys": ["虾粉生产线"]
}
```

响应 `data` 中会包含 `虾粉生产线.称重输送带（流量称）`、`虾粉生产线.盘式真空干燥机` 等叶子 key。

---

## 4. `/api/records` —— B 表（台账/列表，可过滤）

参数：`keys` + 可选 `filter`（见 §1.2）。响应 `data[key]` 是记录数组，每条 `{ "filter": {...}, "value": {...} }`。

**单 key 带过滤**

```bash
curl -X POST http://localhost:6632/api/records \
  -H 'Authorization: Bearer <API_TOKEN_APP>' \
  -H 'Content-Type: application/json' \
  -d '{ "keys": ["溯源"], "filter": { "溯源": { "产品批号": "2606AKO01" } } }'
```

**多 key 联动（一次调用，各自过滤）**

```json
{
  "keys": ["溯源", "设备.台账"],
  "filter": {
    "溯源": { "产品批号": "2606AKO01" },
    "设备.台账": { "所属产线": "虾粉生产线" }
  }
}
```

```json
{
  "status": 200,
  "data": {
    "溯源": [ { "filter": { "产品批号": "2606AKO01" }, "value": { "……" } } ],
    "设备.台账": [ { "filter": { "所属产线": "虾粉生产线", "设备名称": "流量秤" }, "value": { "……" } } ]
  }
}
```

> 每个 key 支持的过滤字段以 yaml 里该 key 的 `filter:` 声明为准。

## 5. `/api/series` —— C 表（时序，重点）

参数：`keys` + 可选 `window`（见 §1.3/§1.4）。响应 `data[key]` = `{ "显示": ..., "单位": ..., "values": [ { "time", "value" }, ... ] }`。

```jsonc
{
  "keys": ["船舶.海况.海水温度", "船舶.能耗.累计耗油量"],
  "window": {
    "船舶.海况.海水温度":  { "start": "2026-01-01 00:00:00", "end": "2026-12-31 00:00:00", "points": 12 },
    "船舶.能耗.累计耗油量": { "start": "2026-01-01 00:00:00", "end": "2026-06-25 00:00:00" }
  }
}
```

```json
{
  "status": 200,
  "data": {
    "船舶.海况.海水温度": {
      "显示": "海水温度",
      "单位": "℃",
      "values": [
        { "time": "2026-01-01 00:00:00", "value": 1.8 },
        { "time": "2026-02-01 00:00:00", "value": 2.1 }
      ]
    }
  }
}
```

### 5.1 时间窗的三种模式（由 `start`/`end` 决定，无独立 time 参数）

| 你传的 | 含义 | 返回 |
|--------|------|------|
| `window` 里该 key 缺省，或不传 `start`/`end` | **当前时刻** | 单点（对齐到网格的最近时刻） |
| `start == end` | 该时刻 | 单点 |
| `end > start` | 区间 | 在 `[start, end]` 上**重采样为 N 点**（N 见 `points`） |

- 时间格式支持 `"YYYY-MM-DD HH:mm:ss"` 和 `"YYYY-MM-DD"`；`start`/`end` 成对出现。日期-only 写法中，`start` 补当天 `00:00:00`，`end` 补当天 `23:59:59`，所以 `start == end == "2026-07-14"` 表示 7 月 14 日全天区间。
- **时区**：所有时间按**业务本地时区**理解（默认 `Asia/Shanghai`，北京时间 UTC+8），不是 UTC。
- **网格对齐**：返回的 `time` 会对齐到该 key 的采样网格（如 `*/10 * * * *` = 每 10 分钟）。网格由 yaml 的 `网格` 字段决定，前端只读结果、无需自己算。
- **量语义**（瞬时/累计，见 yaml）影响区间内每桶取样时刻：瞬时取桶代表时刻的值，累计取到该时刻的累计量。前端照 `values` 画图即可。

### 5.2 父系前缀 / 叶子 key

C 表每个可查询数据项都是**独立叶子 key**。三个取数接口都支持按同表父系前缀展开：这个父系前缀不需要在 yaml 里显式注册。

- 查**叶子 key**（如 `船舶.虾粉生产线.流量秤.实时读数`）→ 正常返回 `values`。
- 查**父系前缀**（如 `/api/series` 传 `船舶信息模型`）→ 自动返回该前缀下全部 C 表叶子 key，效果等同于一次传入所有叶子 key。
- 查**父系前缀**（如 `/api/series` 传 `船舶.桁杆泵吸系统`）→ 自动返回该前缀下全部 C 表叶子 key；`/api/value`、`/api/records` 同理只展开各自表的后代。
- 需要发现有哪些可查 key 时，用 `GET /api/keys` 获取叶子 key 列表，不要把 `/api/series` 当发现接口。

```json
{
  "status": 200,
  "data": {
    "船舶信息模型.船舶方位-经度": {
      "显示": "船舶方位-经度",
      "单位": "°",
      "values": [
        { "time": "2026-06-01 16:15:00", "value": -57.705 }
      ]
    },
    "船舶信息模型.航速": {
      "显示": "航速",
      "单位": "kn",
      "values": [
        { "time": "2026-06-01 16:15:00", "value": 1.635 }
      ]
    }
  }
}
```

**前端正确姿势**：已知要取整组时，可以直接把父系前缀放进 `keys`；已知具体数据项时，直接传叶子 key。全局 `window` 会应用到展开后的所有叶子 key；逐 key `window` 若挂在父系前缀上，也会应用到该前缀展开后的所有叶子 key。

#### 5.2.1 设备运行参数示例：直接蒸煮器

当你问「直接蒸煮器的所有设备运行参数」时，先从 yaml/`/api/keys` 找到与该设备有关的**叶子 key**，再把这些叶子 key 全部放进 `keys`。如果这些叶子 key 有共同父系前缀，也可以直接传父系前缀，由 `/api/series` 自动展开。

当前已有的直接蒸煮器运行态相关叶子 key 是：

- `船舶.虾粉生产线.直接蒸煮器.状态`：设备状态（离散时序）。
- `船舶.虾粉生产线.工艺读数.蒸煮温度.直接蒸煮器`：蒸煮温度（工艺读数时序）。

取当前时刻单点：

```bash
curl -X POST http://localhost:6632/api/series \
  -H 'Authorization: Bearer <API_TOKEN_APP>' \
  -H 'Content-Type: application/json' \
  -d '{
    "keys": [
      "船舶.虾粉生产线.直接蒸煮器.状态",
      "船舶.虾粉生产线.工艺读数.蒸煮温度.直接蒸煮器"
    ]
  }'
```

取同一时间区间时，优先使用全局 `window`：

```json
{
  "keys": [
    "船舶.虾粉生产线.直接蒸煮器.状态",
    "船舶.虾粉生产线.工艺读数.蒸煮温度.直接蒸煮器"
  ],
  "window": {
    "start": "2026-07-14 00:00:00",
    "end": "2026-07-14 23:59:59",
    "points": 20
  }
}
```

不推荐写法：

```json
{
  "keys": ["船舶.虾粉生产线.工艺读数"],
  "window": {
    "船舶.虾粉生产线.工艺读数": {
      "start": "2026-07-14 00:00:00",
      "end": "2026-07-14 23:59:59",
      "points": 20
    }
  }
}
```

这会把 `船舶.虾粉生产线.工艺读数` 整组展开，返回该工艺读数前缀下全部叶子 key，而不只是“直接蒸煮器”相关数据项。需要设备相关数据项时，优先使用能精确覆盖目标范围的父系前缀；如果没有这样的共同前缀，就传明确叶子 key。

#### 5.2.2 三方原文口径示例

当前 registry 同时保留内部业务口径（如 `工厂.虾油提取生产线.*`）和三方文档原文口径（如 `虾油提取生产线.*`、`虾油生产线（生产数据）.*`）。新页面若按三方文档章节组织，优先使用三方原文口径父系前缀：

```json
{
  "keys": [
    "船舶信息模型",
    "天气、海况、水文气象信息",
    "虾粉生产线",
    "虾油提取生产线",
    "蛋白肽生产线（生产数据）"
  ],
  "window": {
    "start": "2026-07-14 00:00:00",
    "end": "2026-07-14 23:59:59",
    "points": 20
  }
}
```

这些父系前缀会展开成各自 C 表叶子 key。常用父系前缀包括：

| 页面/模块 | 表 | 推荐 keys |
|---|---|---|
| 船舶基础方位 | C | `船舶信息模型` |
| 天气海况水文 | C | `天气、海况、水文气象信息` |
| 船舶捕捞产量参数 | A | `船舶捕捞` |
| 桁杆泵吸/传统拖网运行参数 | C | `桁杆连续泵吸捕捞系统【主要参数】`、`传统拖网捕捞系统【主要参数】` |
| 船载虾粉/冻虾/虾肉产量信息 | A | `虾粉生产线信息`、`冻虾生产线信息`、`虾肉生产线信息` |
| 船载虾粉/冻虾/虾肉设备性能 | A | `虾粉生产线`、`冻虾生产线`、`虾肉生产线` |
| 船载虾粉/冻虾/虾肉运行态 | C | `虾粉生产线`、`冻虾生产线`、`虾肉生产线` |
| 船载产品检测 | C | `全脂虾粉产品检测数据（原表）`、`冻虾产品检测数据`、`虾肉产品检测数据` |
| 陆地虾油/蛋白肽主要参数 | A | `虾油提取生产线信息`、`蛋白肽生产线信息` |
| 陆地设备性能 | A | `虾油提取生产线`、`蛋白肽生产线` |
| 陆地设备运行态/检测 | C | `虾油提取生产线`、`蛋白肽生产线`、`成品虾油产品检测参数`、`脱脂虾粉产品检测`、`蛋白肽产品检测参数` |
| 陆地生产数据 | C | `虾油生产线（生产数据）`、`蛋白肽生产线（生产数据）` |

> 注意：同一次 `/api/series` 只能放 C 表 key；A 表主要参数和设备性能参数要走 `/api/value`。

### 5.3 派生值 / 占位 / 真值采集

- **派生**（如各线得率）：响应多一个 `"派生": "<表达式>"` 字段，`values` 是算好的值，前端照常用。
- **占位项**：极少数未配规则的 key，`data[key]` 返回 `{ "note": "占位项，未配置规则..." }`，没有 `values`。
- **真值采集待对接**（yaml 标 `目标成熟度: 真值采集`，如 `船舶.航行` 等）：当前用 fallback 规则出数，响应带 `"note": "真值采集·当前用 fallback 规则（待真实源就绪切换）"`。**接口和字段将来不变**，前端可照现在写。

---

## 6. 错误与排查

| 现象 | 含义 | 处理 |
|------|------|------|
| `data[key] = { "error": "key 未注册" }` | 该 key 不在（内存）注册表 | 先查拼写/是否在 yaml；**若 yaml 里明明有**，多半是后端注册表未热加载 —— 让运维调 `POST /api/reload` 或重启后端 |
| `data[key] = { "error": "X 属 Y 表，请用对应接口（A→/value, B→/records, C→/series）" }` | 用错了接口 | 按提示换接口 |
| `400` filter/window 形状错 | `filter` 没按 key 作用域，或 `window` 混用了全局字段和逐 key 配置 | `filter` 改成 `{ "<key>": {...} }`；`window` 在全局形态和逐 key 形态中二选一 |
| `401` | Token 缺失/无效 | 检查 `Authorization: Bearer <token>` |
| CORS 报错 | 来源未放行 | 联系运维加白名单或走代理 |
| 顶层出现 `warnings` | 请求命中了兼容 alias 或 deprecated key | 业务渲染继续读 `data`；按 `warnings[].replaced_by` 迁移到新 key |

> **「key 未注册」但 yaml 里有** 是最常见的困惑：注册表在后端启动时一次性载入内存（`REG = load()`），新增/改动 yaml 后需要 `/api/reload` 或重启才生效。

---

## 7. 运维/自检接口（前端了解即可）

- `GET /api/health`（**公开**，无需 Token）：存活探测，返回 key 总数、按表/成熟度分布、自检是否通过。可用作前端启动前的连通性检查。
- `GET /api/keys`（运维 Token）：列出全部叶子 key 及其 `表 / 接口 / 成熟度 / 显示 / qualified_key`，并透出 `alias_of / deprecated / replaced_by / remove_after` 等迁移元信息。`v1.0.3` 当前共 662 个 key（A244 / B8 / C410）。**需要 key 清单时以此接口 + `registry/*.yaml` 为准。**
- `POST /api/reload`（运维 Token）：重新加载注册表（改完 yaml 后使新 key 生效，无需重启）。

---

## 8. 三步上手（Checklist）

1. **查 key**：在 `registry/*.yaml` 找到目标 key，记下它的**表**（和 B 的 `filter:` 字段 / C 的 `网格:`、`量语义:`）。
2. **选接口 + 组参数**：A→`/api/value`（仅 keys）；B→`/api/records`（keys + filter）；C→`/api/series`（keys + window/points）；带上 `Authorization: Bearer <使用侧Token>`。
3. **取值**：从 `data["<key>"]` 读结果。传父系前缀时，接口会把结果直接展开到各个同表叶子 key 下。
