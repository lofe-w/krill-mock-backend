# 联调变更规范：后端人员规范与前端人员手册

> 适用阶段：前后端已经进入联调，已有前端页面接入本批接口和数据。
>
> 目标：后端可以继续调整配置和数据结构，但不能让前端突然遇到 `key 未注册`、字段消失、类型变化等不可预期错误。

---

## 1. 总原则

联调后，接口契约进入稳定期。后端对 key、字段、结构、类型做任何会影响前端解析的改动，都必须走兼容流程：

1. 先新增新 key / 新字段 / 新结构。
2. 旧 key / 旧字段在迁移期内继续返回原语义数据。
3. 旧 key / 旧字段标记为 `deprecated`，明确替代项和计划删除日期。
4. 通过 `/api/keys`、接口 `warnings`、`config/变更记录.md` 告知前端。
5. 前端完成迁移并过了约定时间后，后端再删除旧契约。

禁止把旧字段值改成 `1234567890`、`"已废弃"` 这类占位值作为默认方案。废弃说明应放在元信息里，不要混进业务数据里。

---

## 2. 变更分级

| 类型 | 示例 | 是否影响前端 | 后端处理方式 |
|---|---|---:|---|
| 兼容新增 | 新增 key、新增字段、新增接口 | 否 | 可直接增加，记录变更 |
| 兼容扩展 | 枚举新增值、对象新增可选字段 | 低 | 提前说明，前端做兜底 |
| 非兼容变更 | key 改名、字段改名、字段类型变化、数组改对象 | 是 | 必须走废弃流程 |
| 删除 | 删除旧 key、删除旧字段、删除枚举值 | 是 | 迁移期结束后才允许删除 |

判断标准：如果前端旧代码不改就可能报错、空白、渲染异常、图表计算异常，都视为非兼容变更。

---

## 3. 后端人员规范

### 3.1 key 改名：使用 `alias_of`

不要直接删除旧 key。旧 key 应作为 alias 指向新 key：

```yaml
- key: 船舶.虾粉线.流量秤.实时读数
  表: C
  alias_of: 船舶.虾粉生产线.流量秤.实时读数
  deprecated:
    since: "1.1.0"
    remove_after: "2026-08-15"
    replaced_by: 船舶.虾粉生产线.流量秤.实时读数
    reason: "产线命名统一为生产线"
```

效果：

- 前端继续查旧 key 时，接口仍返回可用数据。
- 响应里会带 `warnings`，提示旧 key 已废弃。
- `/api/keys` 会暴露 `deprecated`、`alias_of`、`replaced_by`、`remove_after`。

### 3.2 字段改名：新旧字段并存

不要直接把旧字段删掉。迁移期内同时返回旧字段和新字段：

```yaml
value:
  所属工艺流程: 虾粉生产线
  所属产线: 虾粉生产线
fields:
  所属工艺流程:
    deprecated:
      since: "1.1.0"
      remove_after: "2026-08-15"
      replaced_by: 所属产线
      reason: "字段命名统一"
```

字段级废弃说明目前主要通过 `/api/keys`、文档和变更记录告知前端；业务响应中的字段值仍保持真实业务值。

### 3.3 字段类型变化：新增字段承载新类型

不要把同一个字段从数字改成字符串、从对象改成数组、从数组改成对象。

错误做法：

```json
{ "value": [1, 2, 3] }
```

直接改成：

```json
{ "value": { "items": [1, 2, 3] } }
```

推荐做法：

```json
{
  "value": [1, 2, 3],
  "value_v2": { "items": [1, 2, 3] }
}
```

然后把 `value` 标记 deprecated，前端迁移到 `value_v2` 后再删。

### 3.4 删除规则

删除旧 key / 旧字段前必须满足：

- 已在 `config/变更记录.md` 记录废弃信息。
- 已配置 `deprecated.remove_after`。
- 当前日期已超过 `remove_after`。
- 前端确认已迁移或联调负责人确认可以删除。
- 删除前至少提前一次提醒前端。

建议迁移期：

| 变更规模 | 最短保留时间 |
|---|---:|
| 小字段改名 | 1 周 |
| key 改名 / 普通结构调整 | 2 周 |
| 多页面依赖的大结构调整 | 1 个迭代周期 |

### 3.5 变更记录模板

每次影响前端的变更，都在 `config/变更记录.md` 增加：

```md
## 1.1.0（2026-07-14）

### Deprecated

| 旧契约 | 新契约 | 废弃版本 | 计划删除日期 | 迁移方式 |
|---|---|---|---|---|
| 船舶.虾粉线 | 船舶.虾粉生产线 | 1.1.0 | 2026-08-15 | 全局替换 key 前缀 |

### Breaking Changes

本版本无立即破坏性变更；旧 key 保留到 2026-08-15。

### Frontend Checklist

1. 搜索旧 key / 旧字段。
2. 替换为新 key / 新字段。
3. 验证页面图表、列表、详情弹窗。
4. 确认接口响应不再出现对应 deprecated warning。
```

## 4. 前端人员手册

### 4.1 接口数据仍从 `data` 读取

业务数据仍然从现有位置读取：

```json
{
  "status": 200,
  "data": {
    "船舶.信息": {}
  }
}
```

后端不会把废弃说明混入业务字段值里，因此前端展示层不需要解析 `"已废弃"` 这类假值。

### 4.2 关注 `warnings`

当请求命中了已废弃 key 或 alias key，响应会额外出现 `warnings`：

```json
{
  "status": 200,
  "data": {
    "船舶.虾粉线.流量秤.实时读数": {
      "单位": "kg",
      "values": []
    }
  },
  "warnings": [
    {
      "type": "deprecated_key",
      "key": "船舶.虾粉线.流量秤.实时读数",
      "since": "1.1.0",
      "remove_after": "2026-08-15",
      "replaced_by": "船舶.虾粉生产线.流量秤.实时读数",
      "message": "该 key 已废弃，请迁移到新 key"
    }
  ]
}
```

建议前端在开发环境做统一处理：

- 控制台输出 `warnings`。
- 联调环境可在调试面板显示 deprecated key。
- CI 或页面巡检中统计是否仍使用 deprecated key。

### 4.3 使用 `/api/keys` 做接入自检

前端可定期或启动时读取 `/api/keys`，检查自己使用的 key 是否已经废弃：

```json
{
  "key": "船舶.虾粉线.流量秤.实时读数",
  "表": "C",
  "接口": "/api/series",
  "deprecated": true,
  "alias_of": "船舶.虾粉生产线.流量秤.实时读数",
  "replaced_by": "船舶.虾粉生产线.流量秤.实时读数",
  "remove_after": "2026-08-15"
}
```

如果发现 `deprecated: true`，应尽快替换到 `replaced_by` 或 `alias_of` 指向的新 key。

### 4.4 前端兜底建议

前端解析接口时建议遵守：

- 对新增字段保持兼容，不因多字段报错。
- 对枚举值做默认展示，不假设枚举永远固定。
- 对 `data[key].error` 做可读提示。
- 对列表、对象、数字、字符串的结构变化保持警惕；一旦看到 warning，优先迁移。
- 不依赖字段顺序。

---

## 5. 双方联调约定

后端每次准备改接口契约前，应在联调群或任务记录中说明：

- 旧 key / 旧字段是什么。
- 新 key / 新字段是什么。
- 是否已兼容保留旧契约。
- 计划删除日期。
- 前端需要改哪些页面或模块。

前端每次迁移后，应反馈：

- 已替换的 key / 字段。
- 涉及页面。
- 是否仍看到 deprecated warning。
- 是否可以进入删除倒计时。

---

## 6. 当前项目落地方式

本项目已约定：

- `alias_of`：旧 key 指向新 key，旧请求继续可用。
- `deprecated`：声明废弃版本、计划删除日期、替代项、原因。
- `/api/keys`：暴露 key 级和字段级废弃元信息。
- `warnings`：仅当请求命中废弃 key / alias key 时返回；当前未配置废弃项时，现有响应形状不变。
- `config/变更记录.md`：记录所有影响前端的变更和迁移指引。

当前不会改动既有 registry 配置，因此已经接入的前端不会因为本规范落地而受到影响。
