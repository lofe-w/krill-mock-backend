## 1. 背景与目标

### 1.1 背景

前端为磷虾船项目的可视化大屏提出了数据接口需求（《【磷虾船及工厂】数据对接 API 文档》），涵盖溯源、物流、磷虾船当天/历史、连云港工厂当天/历史共六大模块。原文档以"前端视角"罗列了六个胖接口，把存储、单位、图表结构、聚合口径全部耦合在一起。

本系统**只采纳该文档中的业务范围与数据语义**，在架构上完全重新设计：用一套通用、可配置、可复现的 Mock 后端，简单快速地满足当前业务，并保证将来可低成本地增删改指标与接口。

### 1.2 系统定位

这是一个**确定性的数据 Mock 后端**：事实数据由规则即时计算产生（假数据），维度数据由配置文件维护。系统现阶段不接入真实传感器，但在取数链路上为"未来部分指标接真实数据"预留了扩展位。

### 1.3 设计目标

- **简单、快**：无数据库、无常驻造数进程、无状态，启动即用。
- **通用、灵活、可配置**：指标、维度、规则、渔季、编号规则全部由配置驱动；增删改指标/接口不动框架代码。
- **可复现**：同一时刻的数据由纯函数确定，历史数据任意时间查询结果一致，便于演示、截图、对账。
- **关注点分离**：后端只产出"事实数据本身"，单位后缀、ECharts 结构等展示层逻辑全部交给前端。

### 1.4 非目标（现阶段明确不做）

- 不实现原文档要求的六个胖接口，也不做兼容它们的"组合模板层"。
- 不做展示层（不拼 ECharts option、不把单位拼进数值字符串）。
- 不做 admin / 控制接口（重置种子、注入数据等）。
- 不接真实传感器（仅预留扩展位）。
- 不做鉴权、分页、限流（演示系统不需要）。

## 2. 核心设计理念

### 2.1 懒计算 · 规则即真相

事实数据不落库、不缓存，**查询时按规则即时计算**。每个指标在任意时刻的值定义为一个纯函数：

```
value(指标, t) = f(seed(指标, t_grid))
```

其值只依赖"指标"和"对齐到网格的时刻 t_grid"，与查询发生的当前时间无关。由此：

- **历史一致 / 可复现**：查"6 月 1 日~5 日"，今天查、明天查、明年查，结果逐点完全一致。
- **无中间状态**：没有"回填到一半""写入时钟偏移""两次造数对不上"这类问题——这是它比"造数落库"更稳的根本原因。
- **任意回填**：任意历史区间随查随算。

> 诚实前提：一致性是在"规则不变"的前提下成立的。若修改了某指标的规则，则该指标整段历史会随之重算（演示场景中通常正是期望行为）。若需冻结某段历史不随规则变动，使用 overrides（见 2.4）。

### 2.2 存储与展现分离

后端只返回**纯数值 + 必要的结构化元数据**（数据类型、单位）。单位后缀（`"25℃"`）、ECharts 结构、图表类型与坐标轴、百分比字符串等**全部是前端的展示职责**。前端拼图所需的元数据（单位、图表类型、显示名等）由 `meta` 接口一次性提供。

### 2.3 事实 / 维度 二分

整个系统只有两类数据：

- **事实（Fact）**：随时间变化、可由规则计算的数值序列。例：温度、产量、能耗、航速、库存量。→ 走 `fact` 接口。
- **维度（Dimension）**：人工编排的主数据，不可计算。例：人员、设备、专利、监控地址、商品、仓库、溯源批次图。→ 走 `dim` 接口。

### 2.4 取数三级回退

每个事实值的解析顺序固定为：

```
1. override 命中？     → 返回配置中的覆盖/冻结值      （剧情注入 & 历史冻结）
2. 该指标标记真实源且可用？→ 拉真实数据                  （现阶段为空，未来扩展位）
3. 否则                → 按规则计算                     （函数底座）
```

`overrides` 一物两用：既能为演示注入"剧情"（如某天某设备故障），也能把某段历史"钉死"为固定值。

### 2.5 无数据库 · 无状态

系统不存在"运行时被修改且需重启后保留"的可变状态：无写接口、无 admin、无用户数据、无昂贵到必须缓存的计算（累计型走闭式求值，O(1)）。因此**不引入任何数据库**。全部数据来自：

- 配置文件（指标注册表、维度数据、overrides、渔季、编号规则）→ 启动时加载进内存。
- 事实值 → 纯函数即时计算。

## 3. 系统架构

### 3.1 分层

```
┌─────────────────────────────────────────────────────────────┐
│  接口层 (FastAPI)                                            │
│    POST /api/fact    POST /api/dim    GET /api/meta          │
└───────────────┬─────────────────────────────────────────────┘
                │
┌───────────────▼─────────────────────────────────────────────┐
│  服务层                                                      │
│    FactService   DimService   MetaService                   │
└───────────────┬─────────────────────────────────────────────┘
                │
┌───────────────▼─────────────────────────────────────────────┐
│  取数解析器 Resolver                                         │
│    override → (未来)真实源 provider → 规则                   │
└───────────────┬─────────────────────────────────────────────┘
                │
┌───────────────▼─────────────────────────────────────────────┐
│  造数层 Generators（规则函数库，纯函数）                     │
│    基线_季节_噪声 / 累计 / 轨迹 / 离散状态 / 百分比 / ...     │
└───────────────┬─────────────────────────────────────────────┘
                │
┌───────────────▼─────────────────────────────────────────────┐
│  配置层（启动加载进内存的 Registry）                         │
│    指标注册表 · 维度库 · overrides · 渔季 · 编号规则          │
│    数据源：config/*.yaml                                     │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 数据流

- **fact 请求**：解析时间语义（点/区间/即时）→ 对齐网格 → 对每个 key 调 Resolver 逐点取值 → 组装 raw 响应。
- **dim 请求**：按 `类型 + filter` 从维度库筛选记录 → 组装（含人员时序属性、溯源嵌套结构）→ 返回。
- **meta 请求**：返回指标目录与渔季配置。

### 3.3 一致性与可复现性原理

```python
def seed(key: str, t_grid: str) -> int:
    # 稳定哈希：同一 (指标, 网格点) 永远得到同一随机流
    return int(hashlib.sha256(f"{key}|{t_grid}".encode()).hexdigest(), 16) & 0xFFFFFFFF
```

- 查询时刻 `t` 先按指标的网格粒度（如 3min）向下取整为 `t_grid`。
- 以 `seed(key, t_grid)` 初始化随机流，规则函数据此产出数值。
- 因 seed 只取决于 key 与 t_grid，结果天然确定、可复现。

## 4. 数据模型

### 4.1 指标（事实）模型

每个指标是注册表中的一条声明，核心属性：

| 属性 | 说明 |
|---|---|
| key | 全限定名，点号路径：`实体[.子实体].指标`，如 `磷虾船.海水温度`、`磷虾船.虾肉生产线.虾肉产量` |
| 种类 (kind) | `瞬时`（如温度、航速）/ `累计`（如累计捕捞量、剩余油量）。用于造数与前端窗口计算，**不进 fact 出参** |
| value_type | 值的数据类型：`Number` / `String` / `Boolean`。进 fact 出参的 `type` 字段 |
| unit | 单位（如 `℃`、`kg`、`%`）。作为元数据，进 fact 出参与 meta |
| grid | 造数网格粒度（如 `3min`、`1h`、`1d`） |
| 规则 (rule) | 指向造数层某个规则函数及其参数（见第 6 章） |
| 图表 (chart) | 展示元数据（图表类型、轴名），仅经 meta 暴露给前端 |

**瞬时型 vs 累计型**：

- 瞬时型：`value(t)` 即该时刻读数；点查取最近网格点，区间取序列。
- 累计型：`value(t) = C(t)`，一条单调累计曲线。前端需要"当天/当月/当前渔季/年度"等任意窗口的累计量时，传该窗口的起止时间，结果 = 序列末值 − 首值（前端两个数相减即可，无需后端聚合，也不传海量点）。

> 这正是 fact 接口入参只需 `keys + start + end`、无需"聚合/多窗口"参数的原因：聚合口径被吸收进了"指标定义"，而非"查询参数"。

### 4.2 维度模型

维度是人工维护的主数据，全部由配置文件提供：

| 维度类型 | 关键字段 | 备注 |
|---|---|---|
| 人员 | 人员名称、电话、职务、所在位置、类型、是否在岗 | 职务/所在位置/类型/是否在岗为**时序属性**（变更数组）；姓名/电话为标量 |
| 设备 | 设备名称、所属工艺流程、所属工艺阶段、设备状态、设备温度 | 扁平记录 |
| 专利 | 专利名称、专利编号 | |
| 监控 | 监控名称、监控地址 | CCTV / 视频会议地址 |
| 商品 | 商品编号、批次编号、类别 | 编号 → 类别的映射依据 |
| 仓储（物流） | 货物名称、批号、数量、规格、保质期、生产/入库/出库日期、冷藏状态 | 按批次号查"当前"仓储记录 |
| 溯源 | 类别、溯源链、原料来源追溯、产品检验 | 按批次号查；原料来源可**递归**嵌套 |

**时序属性结构**（人员）：

```jsonc
"职务": [
  { "time": "2026-01-01 00:00:00", "value": "操作工" },
  { "time": "2026-04-01 00:00:00", "value": "班长" }
]
```

**溯源批次图**：批次（batch）之间通过"原料批次"指针形成有向关系，溯源记录中的 `原料来源追溯` 可递归展开其上游原料的溯源信息。该图小且静态，以配置表达、内存遍历。

### 4.3 文档数据原语 → 本系统映射

原文档的多种"花式结构"在本系统中归并为少数原语，全部由上述两类模型 + 前端展示层覆盖：

| 文档原语 | 本系统处理 |
|---|---|
| 标量+单位 `"2.5℃"` | fact 返回 `value=2.5` + `unit="℃"`，前端拼接 |
| 百分比 `"35.5%"` | 建模为独立的 `Number` 指标，`unit="%"` |
| 趋势对象（ECharts） | fact 返回 raw `values` 序列，前端拼 ECharts；图表类型/轴名来自 meta |
| 多周期聚合 `{当天,当月,当前渔季,年度}` | 累计型指标 + 前端传窗口起止、末值减首值 |
| 时序变更数组（人员属性） | dim 维度的时序属性 |
| 对象列表（设备/监控/航迹/库存列表） | dim 维度记录数组 / fact 序列 |
| 实体快照（人员/设备对象） | dim 维度记录 |
| 溯源图谱（递归原料） | dim 溯源类型，嵌套展开 |

### 4.4 overrides 模型

```jsonc
overrides:
  - key: "磷虾船.虾肉生产线.设备状态"   // 可针对事实指标
    start: "2026-06-20 00:00:00"
    end:   "2026-06-20 23:59:59"
    value: "故障"                       // 该时间窗内强制返回此值
```

命中时直接返回，跳过规则计算。

  # —— 累计型 ——

## 6. 造数层：规则函数库

### 6.1 规则函数约定

每个规则是一个**纯函数**，签名统一：

```python
def rule_fn(params: dict, key: str, t_grid: datetime) -> Number | str | bool:
    """根据参数、指标 key、对齐后的时刻，确定性地返回一个值。"""
```

规则通过 `kind` 字段注册到一张分发表 `GENERATORS: dict[str, rule_fn]`。**新增一种数据形态 = 写一个函数 + 注册名字**，框架其余部分不动。

### 6.2 内置规则 kind

| kind | 适用 | 计算方式 |
|---|---|---|
| `基线_季节_噪声` | 瞬时数值（温度、流量、得率…） | `基线 + 季节项(t) + 种子噪声`，可钳制区间 |
| `累计` | 累计量（捕捞量、耗油、产量…） | 闭式：`C(t) = 速率基线 × 网格数(起点→t) + 季节漂移(t) + 种子微扰(t)`，O(1) |
| `轨迹` | 经纬度/航向/航速 | 由路径函数生成坐标序列 |
| `离散状态` | 枚举字符串（设备/绞车状态） | 按权重的种子抽样 |
| `百分比` | 率值（可用 `基线_季节_噪声` 代替） | 同瞬时，`unit="%"` |

### 6.3 种子与确定性

见 3.3。所有规则的随机性都源自 `seed(key, t_grid)`，因此完全可复现。

### 6.4 累计型 O(1) 求值

累计型**不做"把每个网格增量累加 N 次"**，而是把 `C(t)` 定义为可直接求值的闭式函数：

```
C(t) = 速率基线 × 网格数(起点, t) + 季节漂移(t) + 种子微扰(t)
```

任意 `t` 一次算出，单调、可复现；"某窗口累计量" = `C(end) − C(start)`。

## 7. 取数解析器（Resolver）

### 7.1 三级回退

```python
def resolve(key: str, t_grid: datetime):
    if (ov := overrides.hit(key, t_grid)) is not None:
        return ov                          # ① 覆盖/冻结
    # ② 未来真实源 provider（现为空）
    rule = registry[key].rule
    return GENERATORS[rule.kind](rule.params, key, t_grid)   # ③ 规则
```

未来某指标接真实数据时，仅为它注册一个真实源 provider（自带存储），第②级即生效，`fact/dim/meta` 与规则层均不改动。

### 7.2 时间语义

`fact` 的 `start` / `end` 为日期时间字符串（`YYYY-MM-DD HH:mm:ss`），**左闭右闭**：

| 入参 | 语义 | 返回 |
|---|---|---|
| `start == end`（同一时刻） | 点查 | 每个 key 返回 ≤ 该时刻的最近网格点（单元素 `values`） |
| `start != end` | 区间查询 | 每个 key 返回区间内全部网格点序列 |
| `start`、`end` 均空 | 即时 | 取 ≤ now 的最近网格点（单元素 `values`） |

> 超长区间的点数控制（采样/最大点数）现阶段不实现；将来可加可选参数 `max_points`，不破坏现有契约。

## 8. 接口契约

### 8.1 通用约定

**命名约定（全系统强制）**

- **协议/结构字段 → 英文**：`status`、`data`、`keys`、`filter`、`start`、`end`、`values`、`time`、`value`、`unit`、`type`。
- **业务名（指标名、维度类型名、业务字段名）→ 中文**（遵循原文档）：`海水温度`、`溯源`、`仓储`、`货物名称`、`职务` 等。
- 时序点的包装结构统一为 `{ "time": ..., "value": ... }`（英文，结构字段），无论出现在 fact 还是 dim 的人员时序属性中。

**统一响应包络**

```jsonc
{ "status": 200, "data": { /* 业务数据 */ } }
```

**状态码**

| status | 含义 |
|---|---|
| 200 | 成功 |
| 400 | 请求参数错误（缺参/格式错） |
| 404 | 未找到数据（编号不存在等） |
| 500 | 服务端异常 |

请求方式统一 `POST`（`meta` 为 `GET`），`Content-Type: application/json`，UTF-8。

### 8.2 `POST /api/fact` — 事实数据

**入参**

```jsonc
{
  "keys": ["磷虾船.海水温度", "磷虾船.累计捕捞量"],  // 全限定 key，单个或数组
  "start": "2026-06-18 00:00:00",                   // 可空
  "end":   "2026-06-18 23:59:59"                    // 可空
}
```

**出参**（raw；`data` 的 key 原样回显请求 key；`type` 为值的数据类型）

```jsonc
{
  "status": 200,
  "data": {
    "磷虾船.海水温度": {
      "type": "Number", "unit": "℃",
      "values": [ { "time": "2026-06-18 00:00:00", "value": 2.4 }, /* ... */ ]
    },

      "type": "Number", "unit": "kg",
      "values": [ { "time": "2026-06-18 00:00:00", "value": 150000.0 }, /* ... */ ]
    }
  }
}
```

- 点查 / 即时 时，`values` 为单元素数组。
- `type` ∈ `Number` / `String` / `Boolean`；`unit` 来自注册表（无单位为 `""`）。

## 9. 端到端样例

以"磷虾船海水温度，查 2026-06-18 当天"为例，串联从配置到出参的全过程。

**① 配置（metrics.yaml）**

```yaml
- key: 磷虾船.海水温度
  种类: 瞬时
  value_type: Number
  unit: ℃
  grid: 3min
  rule: { kind: 基线_季节_噪声, 基线: 2.5, 噪声: { 分布: 正态, 标准差: 0.3 }, 区间: [-2, 30] }
```

**② 请求**

```jsonc
POST /api/fact
{ "keys": ["磷虾船.海水温度"], "start": "2026-06-18 00:00:00", "end": "2026-06-18 23:59:59" }
```

**③ 计算（服务层伪代码）**

```python
spec = registry["磷虾船.海水温度"]
points = []
for t in grid_range("2026-06-18 00:00:00", "2026-06-18 23:59:59", spec.grid):  # 每3min一个点
    v = resolve("磷虾船.海水温度", t)        # override→真实源→规则
    points.append({"time": fmt(t), "value": round(v, 2)})
```

`resolve` 命中规则 `基线_季节_噪声`：以 `seed("磷虾船.海水温度", t)` 取正态噪声，`2.5 + 噪声`，钳制到 `[-2, 30]`。同一 `t` 永远得到同一 `v`。

**④ 出参**

```jsonc
{
  "status": 200,
  "data": {
    "磷虾船.海水温度": {
      "type": "Number", "unit": "℃",
      "values": [
        { "time": "2026-06-18 00:00:00", "value": 2.41 },
        { "time": "2026-06-18 00:03:00", "value": 2.55 }
        /* ... 当天每 3min 一个点 */
      ]
    }
  }
}
```

**⑤ 前端**：拿到 raw 序列，结合 meta 的 `chart` 与 `unit` 自行拼 ECharts 折线图、给数值加 `℃` 后缀。若需"当天均温"等，自行在序列上计算。

## 11. 扩展性

- **增删改指标**：编辑 `metrics.yaml`，不动代码。
- **新增数据形态**：在 `generators/` 写一个规则函数并注册 `kind`。
- **增删改维度**：编辑 `dimensions.yaml`。
- **新增接口/对外服务**：在现有 `fact / dim / meta` 之上即可满足绝大多数；确需新接口时，按同样的"服务 + 路由"模式增加，复用 Resolver 与造数层。
- **接入真实数据**（未来"两者皆有"）：为目标指标注册真实源 provider（自带存储），Resolver 第②级生效，其余零改动。
